# Technical Review — Issue #2349 Overview

## Verdict: APPROVE-WITH-FIXES

The overview is substantially correct and much stronger than a name-only inventory: it correctly identifies the current self-inflicted neutral-to-Google-to-neutral round trip, the `clientContract.ts` payload problem, the structural `ContentConverters.toGeminiContents()` bypass, side-channel retirement, direct-message path, public usage-metadata ambiguity, and a parser-based gate broad enough to avoid the rejected #2424 pattern. I verified the headline counts and most cited claims against the tree. However, it still has several material gaps and a few inaccuracies that should be fixed before it is treated as the definitive architecture map for planning. The most important issues are: the test-file inventory is incomplete, the production structural inventory undercounts several `.parts`/AFC/usage sites even though the proposed gate would catch them, a few import-symbol claims are incomplete, and the enforcement gate as written risks banning legitimate test characterization unless tests are explicitly scoped.

## Critical findings

None. I did not find a defect that invalidates the overall architecture analysis. The central architectural thesis is evidence-grounded: `RuntimeProvider.generateChatCompletion()` returns `AsyncIterableIterator<IContent>` (`packages/core/src/runtime/contracts/RuntimeProvider.ts:77-84`), `RuntimeGenerateChatOptions.contents` is `IContent[]` (`packages/core/src/runtime/contracts/RuntimeProviderChat.ts:49-67`), `StreamProcessor` converts that stream to `GenerateContentResponse` (`packages/agents/src/core/StreamProcessor.ts:488-510`, `:618-655`), and `TurnProcessor` wraps those responses back to neutral chunks (`packages/agents/src/core/TurnProcessor.ts:253-260`).

## Major findings

### Major 1 — Test inventory excludes three additional test files with structural Gemini converter usage

- **Location in doc:** §3.3, §9.1-9, Appendix A.5 (`overview.md:374-389`, `:653`, `:806-871`)
- **Problem:** The document states that the exact agents test/helper import surface is 54 files and equates test migration with raw `@google/genai` importers. That raw-import count is correct, but it misses agents tests that do not import `@google/genai` and still exercise or assert on structural Gemini `{role, parts}` conversion. Because the issue’s failure mode is explicitly structural, not just import-based, the test inventory is incomplete for the purpose of behavioral migration.
- **Evidence:**
  - The document’s Appendix A.5 command finds 54 raw `@google/genai` test/helper importers; I reproduced the same 54-file list.
  - But `packages/agents/src/core/__tests__/boundaryRecovery.test.ts:59-62` performs a structural round trip with `ContentConverters.toGeminiContents(contents)` and `ContentConverters.toIContents(...)` without importing `@google/genai`.
  - `packages/agents/src/core/chatSession.thinking-toolcalls.repro.test.ts:118-125` converts curated neutral history to Gemini content and inspects `c.role === 'model'` and `c.parts`.
  - The same file again does this at `packages/agents/src/core/chatSession.thinking-toolcalls.repro.test.ts:542-563`, asserting `thoughtSignature` on a Gemini `parts` element before converting back to `IContent`.
- **Required fix:** Add a separate test structural-inventory subsection, not just raw-importer counts. Inventory at least these two additional test files (three converter call sites), and adjust acceptance §9.1-9 from “54 test importers” to “54 raw importers plus structural Gemini fixtures/converter tests.” Make clear which structural tests should be retained as boundary/converter characterization tests and which should be rewritten away from internal `GenerateContentResponse`/`Content` fixtures.

### Major 2 — Anonymous/structural production inventory undercounts important `.parts`, AFC, and usage-metadata internal sites

- **Location in doc:** §2A.4 summary and tables; §9.1-2 (`overview.md:208-266`, `:642-645`)
- **Problem:** The document presents §2A.4 as a “COMPLETE inventory” of anonymous structural Google-envelope literals/mutators and summarizes the surface as 3 synthetic-response fabricators, 5 contract/public builders, 6 history/write-path builders, 5 hook adapters, and 7 subagent/executor sites. That inventory is directionally useful but not complete enough for a planner because it omits several production internal `.parts` readers/mutators and Google-named metadata fields that are not just incidental: they participate in history, AFC, token accounting, and direct-response finalization. The proposed gate would catch many of these (`.parts` on non-neutral values; Gemini usage keys), so the document should inventory/disposition them instead of only relying on gate prose.
- **Evidence:** Examples omitted or underrepresented by the “complete” table:
  - `packages/agents/src/core/client.ts:430-450` mutates `newContent.parts` to strip `thoughtSignature` in `setHistory()`. §2A.4 lists `client.ts:667-668` but not this `.parts` mutation.
  - `packages/agents/src/core/clientHelpers.ts:42-45` checks `content.parts` for `functionResponse`/`functionCall`; `:66` reads `content?.parts`. This is a production Google-content helper surface not listed in §2A.4.
  - `packages/agents/src/core/subagent.ts:563` reads `currentMessages[0]?.parts`; `packages/agents/src/core/subagentNonInteractive.ts:365` reads `currentMessages[0]?.parts`; `packages/agents/src/agents/executor-stream-processor.ts:74` reads `message.parts`. These are subagent/executor structural flows beyond the seven literal builders listed.
  - `packages/agents/src/core/DirectMessageProcessor.ts:859-869` reads `candidate.content?.parts`, creates/falls back to `candidate.content`, and mutates `candidate.content.parts`; §2A.4(d) lists `:865-866` but misses the mutation semantics at `:868-869`.
  - AFC and filtering code still uses `content.parts` as a semantic filter: `packages/agents/src/core/TurnProcessor.ts:728`, `packages/agents/src/core/DirectMessageProcessor.ts:386`, `:764`, and `packages/agents/src/core/hookToolRestrictions.ts:133` filter `Content[]` by `(content.parts?.length ?? 0) > 0`.
  - Internal Google usage keys appear in `packages/agents/src/core/TurnProcessor.ts:844-850` (`promptTokenCount`), `packages/agents/src/core/streamResponseHelpers.ts:149-151`, `:308-314`, and `packages/agents/src/core/MessageConverter.ts:651-662`; §7A and gate check (h) discuss usage metadata, but §2A.4’s “complete” structural inventory does not connect these internal payload sites to required neutral dispositions.
- **Required fix:** Rename §2A.4 unless it is expanded, or expand it into two explicit inventories: (a) literal envelope construction sites, and (b) structural payload access/mutation sites (`.parts`, `candidate.content`, `automaticFunctionCallingHistory`, and Google usage keys) with dispositions. The planner needs to know not only where `{role, parts}` is constructed, but also where existing code semantically depends on the shape.

### Major 3 — Import-symbol inventory has several incomplete symbol lists

- **Location in doc:** §3.2 production inventory table (`overview.md:313-362`)
- **Problem:** The 46-file production importer count is correct, but several rows omit imported Google symbols. This weakens the “complete file inventory” because substitutions and dispositions are partly driven by exact imported symbols.
- **Evidence:** I extracted the actual `@google/genai` import specifiers from the 46 production files. Mismatches include:
  - `core/DirectMessageProcessor.ts`: doc row #14 lists `GenerateContentResponse`, `Content`, `GenerateContentConfig`, `SendMessageParameters`, `PartListUnion`, `ApiError`; the file imports `ApiError` and also imports `GenerateContentResponse` as a value (`packages/agents/src/core/DirectMessageProcessor.ts:7-13`). The doc is acceptable here, but row should note the value import split because `GenerateContentResponse` is used for casts/typing and `ApiError` is runtime.
  - `core/beforeModelHookDecision.ts`: row #20 says `FinishReason` and `GenerateContentResponse` are type-only and no runtime enum replacement is needed. That is true for imports, but the file also casts a hook response with `beforeModelResult.getSyntheticResponse() as GenerateContentResponse` and inspects `candidate.finishReason` (`packages/agents/src/core/beforeModelHookDecision.ts:54-65`). It is not merely a harmless type importer; it is a synthetic-response decision point.
  - `core/streamRequestHelpers.ts`: row #21 omits that `FinishReason` is imported as a runtime value (`packages/agents/src/core/streamRequestHelpers.ts:20`) and used in the structural candidate patch (`:162-169`). The row says `FinishReason (value)` but the table’s symbol list formatting also includes a dangling comma after `SendMessageParameters`; make it exact.
  - `core/streamResponseHelpers.ts`: row #22 lists `FinishReason (value)` correctly, but the table does not call out its internal Google usage-metadata dependency; `streamResponseHelpers.ts:146-151` and `:308-314` read `usageMetadata.promptTokenCount` / `candidatesTokenCount` / `totalTokenCount`.
  - `agents/executor-tool-dispatch.ts`: row #35 splits imports into `Type`, `Content`, `Part`, `FunctionCall`, `Schema`, `FunctionDeclaration`; actual imports include `Type` as a value plus `Schema`/`FunctionDeclaration` from a separate import (`packages/agents/src/agents/executor-tool-dispatch.ts` import extraction), so the table is broadly right but should explicitly identify which are value imports because the runtime enum replacement is an acceptance risk.
- **Required fix:** Regenerate §3.2 from actual import specifiers and annotate value imports versus type-only imports consistently. For rows whose import list is correct but the behavior is more significant than the list implies (`beforeModelHookDecision.ts`, `streamResponseHelpers.ts`), add role/disposition detail so the planner does not miss synthetic-response or usage-metadata dependencies.

### Major 4 — The document overstates `toModelStreamChunk(iContent)` sufficiency for target conversion

- **Location in doc:** §2.4 and §5.3-3 (`overview.md:139-141`, `:515`)
- **Problem:** The document correctly notes that `toModelStreamChunk` does not copy `providerMetadata`, but still presents `toModelStreamChunk(iContent)` as the target conversion in the diagram and says it is sufficient for stop/finish/usage/id. That is true only for `IContent.metadata` fields, not for all data currently available on the synthetic `GenerateContentResponse`. In particular, `streamChunkWrapper.responseToModelStreamChunk` also maps `resp.responseId` and `resp.usageMetadata`, while direct/core adapters can produce `ModelOutput.providerMetadata` from Gemini `promptFeedback`, `safetyRatings`, and `groundingMetadata`.
- **Evidence:**
  - `toModelStreamChunk` copies only `stopReason`/`finishReason`, `usage`, and `id` from `IContent.metadata` (`packages/core/src/llm-types/modelEnvelope.ts:188-210`). It does not copy `providerMetadata` (`:58` defines the slot, but `:188-210` never sets it).
  - Current `streamChunkWrapper.responseToModelStreamChunk` maps `resp.responseId` to both `content.metadata.id` and `chunk.responseId` (`packages/agents/src/core/streamChunkWrapper.ts:91-93`, `:152-153`), maps `resp.usageMetadata` through `usageMetadataToUsageStats` (`:125-127`), and pulls hook restrictions from side channels (`:129-157`).
  - Core’s neutral Gemini adapter preserves provider-specific metadata from real Gemini responses: `packages/core/src/code_assist/contentGeneratorAdapters.ts:195-210` maps `promptFeedback`, `safetyRatings`, and `groundingMetadata` into `ModelOutput.providerMetadata`.
- **Required fix:** Change the target conversion language from “use `toModelStreamChunk(iContent)`” to “use `toModelStreamChunk(iContent)` only after/if provider output metadata is normalized onto `IContent.metadata` or a wrapper extends it.” Add an explicit inventory item for provider metadata beyond AFC (Gemini prompt feedback/safety/grounding, response id source, and any provider-specific raw metadata) and specify whether agents must preserve it, ignore it by design, or keep it at provider-only boundaries.

### Major 5 — Gate scope for tests conflicts with acceptance that no fabricated candidates remain in tests

- **Location in doc:** §8 and §9.1-9 (`overview.md:603-634`, `:653-654`)
- **Problem:** The gate is scoped to production files only, but acceptance says no fabricated `{candidates:[...]}` fixtures remain in agents tests. Those two statements can coexist only if the document describes a separate test gate or a deliberate allowance for boundary/converter characterization tests. As written, the acceptance is not mechanically enforceable and may also be too broad: some tests should probably continue to characterize legacy converter boundaries while the boundary exists.
- **Evidence:**
  - §8 explicitly says the parser gate detects violations under `packages/agents/src` production files, excluding `*.test.*`/`*.spec.*`/`__tests__`/`*-test-helpers*` (`overview.md:607-609`).
  - §9.1-9 requires “no fabricated `{candidates:[...]}` fixtures remain in agents tests” (`overview.md:653`).
  - Existing tests include structural converter tests without raw imports, e.g. `boundaryRecovery.test.ts:59-62` and `chatSession.thinking-toolcalls.repro.test.ts:542-563`, and the 54 raw-import tests likely include many explicit response fixtures.
- **Required fix:** Add a test-enforcement policy. Either (A) extend a separate, narrower test gate to ban `GenerateContentResponse` fixtures except in named converter/boundary tests, or (B) revise acceptance to require behavioral migration of agent-loop tests while allowing explicitly named legacy-boundary characterization tests. Without that distinction, the document is internally inconsistent on test scope.

## Additional open questions

1. **Should converter-boundary tests be allowed to remain structural?** The overview should distinguish “agent-loop tests must not assert `GenerateContentResponse` internals” from “converter/boundary tests may intentionally assert Gemini structural compatibility.” Current tests such as `boundaryRecovery.test.ts:59-62` and `chatSession.thinking-toolcalls.repro.test.ts:542-563` need an explicit disposition.
2. **What provider metadata beyond AFC is in scope?** The overview calls out AFC and `providerMetadata` propagation, but the neutral core adapter also maps Gemini `promptFeedback`, `safetyRatings`, and `groundingMetadata` into `ModelOutput.providerMetadata` (`packages/core/src/code_assist/contentGeneratorAdapters.ts:195-210`). Decide whether agents must preserve these through events/history, or whether they remain provider/core-only.
3. **What replaces Google-shaped `Content[]` in executor prompt configuration?** `PromptConfig.initialMessages?: Content[]` (`packages/agents/src/agents/types.ts:87-95`) and `applyTemplateToInitialMessages(initialMessages: Content[])` (`packages/agents/src/agents/executor.ts:894-898`) look like a public/extension configuration surface, not just internal currency. The overview flags `executor-prompt-builder.ts`, but the planner also needs a compatibility decision for the configuration schema itself.
4. **How should usage-token sync work after response neutralization?** Current streaming/direct code uses `usageMetadata.promptTokenCount` for token sync (`packages/agents/src/core/TurnProcessor.ts:844-850`, `packages/agents/src/core/streamResponseHelpers.ts:149-151`). The target should specify whether this reads `UsageStats.promptTokens`, accumulated `ModelOutput.usage`, or `IContent.metadata.usage` and how it preserves fallback behavior when usage is absent.
