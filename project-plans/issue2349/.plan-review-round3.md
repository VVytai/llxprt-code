# Technical Review: PLAN-20260707-AGENTNEUTRAL

## Verdict: REJECT

The plan is substantially stronger than a source-swap plan: it names the synthetic response round-trip, the two side-channels, the structural `toGeminiContents` bypasses, the cross-package `clientContract.ts` blast radius, and the central allow-list enforcement model. However, it is not executable as written. Several phases require characterization tests to pass against the current tree while asserting future neutral return types that the current code provably does not return; one early phase attempts to remove `StreamProcessor`/`streamRequestHelpers` synthetic-response machinery before the plan's own build-order invariants say the synthetic path can die; and the verification template contains repo-wide fraud/mutation gates that will fail on existing tests or fail to test the migrated files. A coordinator executing this literally will either block, weaken verification, or implement a shape-preserving workaround to satisfy contradictory phase acceptance.

## Critical findings

### Critical 1 â€” P12 is an impossible characterization phase: it must pass against current code while asserting future `ModelOutput`

**Location in plan:** `project-plans/issue2349/plan/12-direct-message-characterization-tdd.md:15-17`, `:34-42`

**Problem:** P12 is labeled a characterization/safety-net phase and requires its tests to pass against current code (`npm test ... # PASS against current code`), but its required assertions are that `DirectMessageProcessor.generateDirectMessage` returns neutral `ModelOutput` and not a `GenerateContentResponse`. That is not current behavior; it is the implementation target for P13. This violates the migration TDD strategy: characterization tests should pin observable current behavior, while RED tests may target new APIs. As written, the phase cannot both pass against current code and assert the future return type.

**Evidence:**

- The phase requires future neutral output: `DirectMessageProcessor.generateDirectMessage` returns neutral `ModelOutput` on both paths and no `GenerateContentResponse` cast/envelope (`project-plans/issue2349/plan/12-direct-message-characterization-tdd.md:15-17`).
- The same phase requires those tests to pass against current code (`project-plans/issue2349/plan/12-direct-message-characterization-tdd.md:34-42`).
- Actual current code returns `GenerateContentResponse`: `generateDirectMessage(...): Promise<GenerateContentResponse>` at `packages/agents/src/core/DirectMessageProcessor.ts:166-169`; additional direct-path helpers also return `Promise<GenerateContentResponse>` at `:258`, `:401`, and `:750`.
- The current blocking path explicitly fabricates a candidate envelope: `_buildBlockingSyntheticResponse(...): GenerateContentResponse` at `packages/agents/src/core/DirectMessageProcessor.ts:677-700`.

**Required fix:** Split this into two coherent gates: (1) characterization tests that pass against current code and assert only observable behavior through the public/direct-message surface (text, usage, AFC, hook filtering, history), without requiring `ModelOutput` as the current return type; and (2) P13 RED/implementation tests (or P13 acceptance) that assert the return type and internal value become `ModelOutput`. Do not label future-type assertions as â€śPASS against currentâ€ť.

### Critical 3 â€” P07 attempts to delete/neutralize synthetic-response hook machinery before the plan says the synthetic path can die

**Location in plan:** `project-plans/issue2349/plan/07-streamprocessor-impl.md:31-38`; conflicts with `project-plans/issue2349/plan/08-turnprocessor-turn-impl.md:71-72` and `project-plans/issue2349/plan/13-directmessageprocessor-impl.md:43-70`

But the current `StreamProcessor` still calls a synthetic-response callback from the before-model hook decision path. The plan later explicitly says `sendMessage` remains `GenerateContentResponse` and `convertIContentToResponse` remains until P13. This is a dependency-ordering contradiction: P07 cannot remove the synthetic response type from `StreamProcessor` while its before-model blocking/hook decision still passes `GenerateContentResponse` objects.

**Evidence:**

- P07 requires `_patchMissingFinishReason` to be neutral and deletes the `{candidates}` + `FinishReason.STOP` path (`project-plans/issue2349/plan/07-streamprocessor-impl.md:34`) and drops `GenerateContentResponse`/`FinishReason` imports from `StreamProcessor` (`:37`).
- Current `StreamProcessor` passes a callback `(resp, candidate) => this._patchMissingFinishReason(resp, candidate)` to `enforceBeforeModelHookDecision` (`packages/agents/src/core/StreamProcessor.ts:365-369`).
- Current `_patchMissingFinishReason` is typed in `GenerateContentResponse` and delegates to `patchMissingFinishReason` (`packages/agents/src/core/StreamProcessor.ts:378-383`).
- Current `streamRequestHelpers.patchMissingFinishReason` exists at `packages/agents/src/core/streamRequestHelpers.ts:162`.

**Required fix:** Move the `patchMissingFinishReason`/before-model blocking synthetic-response deletion to the same phase that removes the final synthetic direct path (P13), or add an earlier explicit neutral replacement for `enforceBeforeModelHookDecision` and `BeforeModelHookOutput.getSyntheticResponse()` before P07 tries to remove the `GenerateContentResponse` types from `StreamProcessor`. P07 should only remove the streaming provider-output round-trip if the before-model synthetic hook path remains typed and isolated until P13.

### Critical 4 â€” The verification-template mock-theater grep is non-executable against the existing repo

**Location in plan:** `project-plans/issue2349/plan/verification-template.md:15-18`; global-gate restatement at `project-plans/issue2349/plan/00-overview.md:84-89`

**Problem:** The shared verification template says verification greps tests for `toHaveBeenCalled|toHaveBeenCalledWith` and fails if found. It does not scope that check to new/modified tests in the phase. The current `packages/agents/src` tree already contains hundreds of legitimate and illegitimate existing `toHaveBeenCalled*` assertions. If the template is followed literally, nearly every NNa phase fails before any implementation starts. If verifiers ignore it, the anti-fraud gate becomes discretionary.

**Evidence:**

- Template says mock-theater grep fails on `toHaveBeenCalled|toHaveBeenCalledWith` (`project-plans/issue2349/plan/verification-template.md:15-18`).

- RULES.md forbids mock interactions as a test target (`dev-docs/RULES.md:102-108`), but a migration plan must distinguish pre-existing tests from tests added or rewritten by this plan.

**Required fix:** Scope fraud checks mechanically to files added/modified by the current phase (or to the P28 migrated test set once rewritten). For pre-existing tests outside a phase's touch set, record them as legacy debt unless P28 explicitly rewrites them. The template must say exactly how to compute the file set (`git diff --name-only` for phase changes, plus listed test files) and must not fail on unrelated existing tests.

## Additional risks / open questions

1. **Large atomic P21 risk remains high.** The plan correctly identifies P21 as atomic, but it is the largest blast-radius phase. Add a generated consumer inventory from preflight output to the P21 phase file, not just prose categories, so no CLI/core consumer is missed.
2. **Hook wire byte-compatibility needs a concrete golden fixture.** The plan says hook JSON wire byte-shape is preserved, but the phase list should name the exact golden test file and fixture for before-model/after-model/before-tool-selection payloads. Otherwise G3 can be preserved structurally but still drift semantically.
3. **`any`/type assertion ban vs legacy structural conversion needs careful enforcement.** The neutral legacy converter is typed on `unknown`, which is good, but phases must explicitly forbid `as` casts in converter implementation. RULES.md forbids type assertions (`dev-docs/RULES.md:78-83`); the plan should require type predicates for every legacy shape branch.

Overall: the plan captures the right architectural target, but it must be revised before execution. The highest-priority fixes are to make characterization phases actually pass against current behavior, move synthetic-response deletion steps to a build-valid order, scope anti-fraud verification to changed tests, and make mutation/requirements traceability mechanically real.
