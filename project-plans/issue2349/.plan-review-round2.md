# Technical Review — PLAN-20260707-AGENTNEUTRAL

## Verdict: REJECT

The plan is directionally aligned with the issue intent and the authoritative overview: it targets structural Google-shape removal rather than a #2424-style import/source swap, names the correct production inventory, preserves the G3 hook-wire exception behind a central allow-list, covers the clientContract/CLI/core blast radius, and generally follows the neutral end-to-end architecture. However, it is not executable as a literal TDD coordination plan yet. Multiple verification phases are malformed against the mandatory phase template, several TDD phases omit required implementation/test-task sections, one verification phase directly contradicts its owning implementation phase on the `sendMessage` return-type sequencing, and the mutation gate is partly impossible for the first core slice because the core package has no Stryker dependency/config while the plan merely says to invent one inside the implementation phase. These are not cosmetic issues: a coordinator executing the plan mechanically would either skip required verification, fail build/verification gates for the wrong reason, or be told to perform contradictory actions.

## Critical findings

### Critical 1 — Many phase files do not satisfy the mandatory PLAN-TEMPLATE section structure

**Location in plan:** many `project-plans/issue2349/plan/*.md` files, especially every `NNa` verification sibling and most characterization/TDD phases.

**Problem:** `dev-docs/PLAN-TEMPLATE.md` requires each phase to contain `Phase ID`, `Prerequisites`, expanded requirements with full behavior, implementation tasks, verification commands, success criteria, failure recovery, and completion marker (`dev-docs/PLAN-TEMPLATE.md:23-41`, `:55-83`, `:185-201`). The shared verification template explicitly says it is “NOT a substitute for the per-phase sections” and that each `NNa` file itself must carry the full phase structure (`project-plans/issue2349/plan/verification-template.md:1-3`). In reality, many verification siblings contain only `Requirement Behavior Verified`, `Checks`, `Holistic Assessment`, and a completion marker; they omit required headings and executable commands/success criteria. Example: `03a-neutral-gap-types-stub-verification.md` has no `## Requirements Implemented (Expanded)`, no `## Implementation Tasks`, no `## Verification Commands`, and no `## Success Criteria` headings (`project-plans/issue2349/plan/03a-neutral-gap-types-stub-verification.md:9-29`). `04a-neutral-gap-types-tdd-verification.md` likewise lacks implementation tasks, verification commands, success criteria, and failure recovery (`project-plans/issue2349/plan/04a-neutral-gap-types-tdd-verification.md:9-27`).

A mechanical section audit also found the same malformed pattern across `05a`, `06a`, `07a`, `08a`, `09a`, `10a`, `11a`, `12a`, `13a`, `14a`, `15a`, `16a`, `17a`, `18a`, `19a`, `20a`, `21a`, `22a`, `23a`, `24a`, `25a`, `26a`, `27a`, `28a`, `29a`, `30a`, `31a`, `32a`, and `33a`, plus several TDD/characterization phases whose test-task sections are not under `## Implementation Tasks`. This violates the method documents and makes autonomous execution brittle.

**Evidence:**
- Template requirements: `dev-docs/PLAN-TEMPLATE.md:23-41`, `:55-83`, `:185-201`.
- Shared template says per-phase sections are still required: `project-plans/issue2349/plan/verification-template.md:1-3`.
- Example malformed NNa: `project-plans/issue2349/plan/03a-neutral-gap-types-stub-verification.md:9-29`.
- Example malformed NNa: `project-plans/issue2349/plan/04a-neutral-gap-types-tdd-verification.md:9-27`.
- Example TDD phase with `## Test Tasks` but no required `## Implementation Tasks`: `project-plans/issue2349/plan/04-neutral-gap-types-tdd.md:37-80`; required section name is `## Implementation Tasks` per `dev-docs/PLAN-TEMPLATE.md:55-80`.

**Required fix:** Rewrite every phase file (including every `NNa`) to conform to the exact required phase template. Do not rely on `verification-template.md` as a replacement for missing per-phase sections. For TDD/characterization phases, either rename `## Test Tasks` to `## Implementation Tasks` or add `## Implementation Tasks` with the concrete test-writing tasks, files, markers, and requirement tags. For each verification sibling, include expanded requirement text, concrete checks/commands under `## Verification Commands`, explicit success criteria, failure recovery, and completion marker contents.

### Critical 2 — Phase 08a contradicts Phase 08 on `TurnProcessor.sendMessage` and would fail a correct implementation

**Location in plan:** `plan/08-turnprocessor-turn-impl.md` and `plan/08a-turnprocessor-turn-impl-verification.md`.

**Problem:** Phase 08 explicitly states that it is streaming-only and must **not** flip `TurnProcessor.sendMessage`; `sendMessage` remains `Promise<GenerateContentResponse>` until Phase 13 (`project-plans/issue2349/plan/08-turnprocessor-turn-impl.md:11-21`, `:53-54`, `:71-74`, `:77-84`). That staging is consistent with the actual current code: `TurnProcessor.sendMessage` currently returns `Promise<GenerateContentResponse>` (`packages/agents/src/core/TurnProcessor.ts:130-133`) and direct-path synthetic callers still exist (`packages/agents/src/core/DirectMessageProcessor.ts:649`, `:677`, `:752`, `:806`, `:853`, `:894`).

But Phase 08a’s checklist requires `sendMessage` to return `ModelOutput` in Phase 08 (`project-plans/issue2349/plan/08a-turnprocessor-turn-impl-verification.md:18`). The same file’s prose correctly says the flip is deferred to P13 (`project-plans/issue2349/plan/08a-turnprocessor-turn-impl-verification.md:11-12`), so the checklist is internally contradictory. A coordinator following the checklist would fail a correct P08 implementation or prematurely flip the direct path before P13.

**Evidence:**
- P08 defers the flip: `project-plans/issue2349/plan/08-turnprocessor-turn-impl.md:11-21`, `:53-54`, `:71-74`.
- P08a contradictory check: `project-plans/issue2349/plan/08a-turnprocessor-turn-impl-verification.md:18`.
- Current code confirms `sendMessage` still returns `GenerateContentResponse`: `packages/agents/src/core/TurnProcessor.ts:130-133`.
- Direct path still has synthetic response machinery: `packages/agents/src/core/DirectMessageProcessor.ts:649`, `:677`, `:752`, `:806`, `:853`, `:894`.

**Required fix:** Change the P08a checklist to verify the intended deferral: `sendMessage` still returns `Promise<GenerateContentResponse>` and the verifier confirms this is expected until P13. Move the `sendMessage returns ModelOutput` check exclusively to P13/P13a, where P13 already correctly specifies the flip (`project-plans/issue2349/plan/13-directmessageprocessor-impl.md:11-19`, `:33-45`, `:59-64`, `:80-86`).

### Critical 3 — The mutation gate is not executable for the core neutral-gap slice

**Location in plan:** `plan/05-neutral-gap-types-impl.md`, `plan/verification-template.md`, package configs.

**Problem:** Phase 05 modifies `packages/core/src/llm-types/*` and requires a scoped Stryker mutation score ≥80% (`project-plans/issue2349/plan/05-neutral-gap-types-impl.md:64-66`). But the core package has no Stryker script/dependencies/config: `packages/core/package.json` defines only build/lint/format/test/typecheck scripts (`packages/core/package.json:474-480`) and no `@stryker-mutator/*` dev dependencies in the visible dependency sections. The repository only has Stryker configs for `packages/agents`, `packages/cli`, `packages/mcp`, and `packages/providers`; there is no `packages/core/stryker.conf.json`. By contrast, `packages/agents/package.json` does have `test:mutation:api` and Stryker dev dependencies (`packages/agents/package.json:30-38`, `:66-73`), and `packages/agents/stryker.conf.json` is scoped to `src/api/**/*.ts` (`packages/agents/stryker.conf.json:3-8`).

The plan acknowledges the problem but punts it into the implementation phase: “the core package has no stryker.conf yet, so pass an explicit `--mutate` and `stryker.conf.json` equivalent OR add a minimal core stryker config as part of this phase” (`project-plans/issue2349/plan/05-neutral-gap-types-impl.md:64-66`). That is not an executable verification command and it adds unplanned tooling/config changes in the same phase as production code. It also conflicts with the plan’s own requirement for preflight dependency/tool verification before implementation (`dev-docs/PLAN.md:338-439`; `project-plans/issue2349/plan/00a-preflight-verification.md:77-83` checks parser/fast-check but not core Stryker).

**Evidence:**
- P05 demands core Stryker with no concrete command/config: `project-plans/issue2349/plan/05-neutral-gap-types-impl.md:64-66`.
- Verification template assumes `packages/agents/stryker.conf.json`: `project-plans/issue2349/plan/verification-template.md:26-36`.
- Core scripts lack mutation tooling: `packages/core/package.json:474-480`.
- Agents package has Stryker tooling: `packages/agents/package.json:30-38`, `:66-73`.
- Agents Stryker config mutates only API files by default: `packages/agents/stryker.conf.json:3-8`.

**Required fix:** Add an explicit pre-implementation tooling phase or preflight gate that makes the mutation tooling executable for core before P05, or scope P05’s mutation gate to an already-configured package with a concrete command. If adding `packages/core/stryker.conf.json` and Stryker dev deps is the intended fix, make that a dedicated phase with TDD/verification and no production-code mixing. Then update P05 with a concrete command that can be copied and run exactly.

### Critical 4 — The plan’s “30% property-based” gate is specified per phase but not actually enforced for multi-file phases

**Location in plan:** `plan/verification-template.md`, TDD phases.

**Problem:** The governing template requires ≥30% property-based tests (`dev-docs/PLAN.md:780-815`) and this plan repeats that gate (`project-plans/issue2349/plan/verification-template.md:25`). However, several phase-specific commands count property tests in only one file even when the phase creates multiple test files. Example Phase 04 creates both `agentMessageInput.test.ts` and `modelEnvelope.afc-providerMetadata.test.ts` (`project-plans/issue2349/plan/04-neutral-gap-types-tdd.md:39-42`) but the verification command counts only `agentMessageInput.test.ts` (`project-plans/issue2349/plan/04-neutral-gap-types-tdd.md:74-80`). That allows the phase as a whole to fall below 30% while still passing the command. The same pattern appears in later phases where only one named test file is counted despite multi-file or multi-behavior slices.

**Evidence:**
- Global 30% requirement: `dev-docs/PLAN.md:780-815`; plan template: `project-plans/issue2349/plan/verification-template.md:25`.
- P04 creates two test files: `project-plans/issue2349/plan/04-neutral-gap-types-tdd.md:39-42`.
- P04 counts only one file: `project-plans/issue2349/plan/04-neutral-gap-types-tdd.md:74-80`.

**Required fix:** For every TDD phase, compute the property-test ratio over **all tests added/modified by the phase**, not one hand-picked file. Add a reusable command or script that takes the phase file list and reports total tests, property tests, and percentage, and require the `NNa` verifier to paste the counts.

### Major 4 — The interim structural grep ratchet is intentionally broad and can fail on known false positives before the AST gate exists

**Location in plan:** `verification-template.md`, `07-streamprocessor-impl.md`.

**Problem:** The overview explains that a naive grep for `candidates`/`parts` is both too broad and too narrow; the final gate must be AST-context-aware to spare domain candidates while catching structural `.parts` mutators (`project-plans/issue2349/overview.md:756-780`). But before P31, the plan uses an interim grep counter over `{ candidates:`, `role:'model'`, `role:'user'`, `.parts`, `toGeminiContents`, and usage keys (`project-plans/issue2349/plan/verification-template.md:37-43`; `project-plans/issue2349/plan/07-streamprocessor-impl.md:38-40`). Because it is a count, not a pass/fail gate, it may still work as a rough trend, but the plan says every migration slice must strictly reduce it or fail (`project-plans/issue2349/plan/verification-template.md:37-43`). A phase could legitimately not touch any broad-grep false positives and fail the “strictly lower” rule, or a phase could delete real structural hits while adding a false positive count elsewhere and fail for the wrong reason.

**Evidence:**
- Overview says naive identifier checks are invalid: `project-plans/issue2349/overview.md:756-780`.
- Interim grep is broad: `project-plans/issue2349/plan/verification-template.md:37-43`.
- P07 establishes it as a strict ratchet: `project-plans/issue2349/plan/07-streamprocessor-impl.md:38-40`.

**Required fix:** Define the interim counter as advisory except for files in the current slice, or implement the AST gate skeleton earlier (before P07) so all ratchet counts use the same context-aware logic. If the strict ratchet remains, list known false positives and subtract them through the same central allow-list mechanism, not ad hoc reasoning.

### Major 5 — The plan is faithful to the overview’s inventory, but the final gate arrives after most migration work

**Location in plan:** P07-P31 sequence.

**Problem:** The plan defers the AST gate implementation to P29-P31 (`project-plans/issue2349/plan/00-overview.md:54-57`; `execution-tracker.md:40-43`), after all production migration slices P07-P27 have already run. The interim grep ratchet is not structurally precise (see Major 4). This means most of the migration proceeds without the actual anti-#2424 enforcement in place, relying on human review and a broad grep. Since this issue’s core failure mode is structural bypass, late gate implementation increases risk that earlier slices accidentally add a bypass and only discover it after much downstream work.

**Evidence:**
- Production migration slices complete by P27: `project-plans/issue2349/plan/execution-tracker.md:18-39`.
- Gate stub/TDD/impl are P29-P31: `project-plans/issue2349/plan/execution-tracker.md:40-43`.
- Overview demands a parser-based structural gate to catch #2424 vectors: `project-plans/issue2349/overview.md:754-795`.

**Required fix:** Move at least the core AST gate skeleton plus `--count` implementation earlier, preferably before P07, with tests for the main structural patterns. It can start with a nonzero baseline and ratchet down, but it should be the same AST-context-aware mechanism used at the end.

## Minor findings

### Minor 1 — Phase numbering is sequential and complete, but the plan starts at P03 without explicit analysis/pseudocode phase artifacts in the execution sequence

The actual files include P03-P33 with no numeric skips and a P0.5 preflight (`project-plans/issue2349/plan/00-overview.md:26-59`; `execution-tracker.md:12-45`). This satisfies the “never skip numbers” spirit for implementation phases, and analysis/pseudocode artifacts already exist. However, `dev-docs/PLAN.md` shows analysis and pseudocode as Phase 01/02 artifacts in the canonical plan structure (`dev-docs/PLAN.md:187-209`, `:443-542`). Because this plan starts at P03, add a short note explaining that P01/P02 were completed out-of-band and are represented by `analysis/domain-model.md` and `analysis/pseudocode/*.md`, not executable phases. This avoids a literal coordinator thinking P01/P02 were skipped.

### Minor 2 — Some line references are expected to drift and are guarded by preflight, but the phase text still relies heavily on exact old line numbers

The plan’s preflight correctly requires fresh pasted output and plan updates if line ranges drift (`project-plans/issue2349/plan/00a-preflight-verification.md:37-61`, `:120-121`). That mitigates many line-number risks. Still, implementation phases repeatedly cite current line numbers in file inventories. Ensure the phase completion markers include the refreshed line evidence from preflight before subagents use those phase files.

### Minor 3 — `npm run format` in final verification modifies the tree

Phase 33 requires `npm run format` (`project-plans/issue2349/plan/33-full-verification.md:17-21`), which is consistent with project memory before check-in, but it is a mutating command. For a final verification phase, also require `git diff --check`/`npm run format:check` or a post-format `git diff` review so the verifier does not accidentally sign off unreviewed formatter changes. Root package scripts include both `format` and `format:check` (`package.json:96-99`).
