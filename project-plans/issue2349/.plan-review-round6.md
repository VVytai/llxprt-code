# Review of PLAN-20260707-AGENTNEUTRAL

## Verdict: REJECT

The plan is unusually thorough and correctly captures much of the architectural intent of issue #2349: the governing principle is structural neutrality, the synthetic response round-trip is not acceptable, `clientContract.ts` must be neutralized cross-package, the usage-metadata boundary is explicitly decided, and the final AST gate is far stronger than a raw-import shrink-ratchet. However, I cannot approve it for mechanical execution because several phase-level contradictions and impossible call patterns would either block execution early or let a coordinator create a mid-plan Google-shaped compatibility island while believing the plan is still compliant. The highest-risk defects are: P02 requires real gate fail-mode verification against fixture files it never creates and that do not exist; P29 contradicts P02 by allowing the cheap #2424 checks to be stubbed back to `[]`; P07 requires passing a neutral `ModelOutput` through `AgentExecutionBlockedError` even though the current constructor is still typed to `GenerateContentResponse` and P07 does not retype it; and P17 contains a literal placeholder pseudocode range where the method docs require concrete numbered-line citations.

### Critical 3 — P07's AfterModel blocking path is not build-green: it requires passing a neutral `ModelOutput` through an error constructor that still requires `GenerateContentResponse`

**Location in plan:** `project-plans/issue2349/plan/07-streamprocessor-impl.md:67-79`, especially line 77

**Problem:** P07 says the streaming AfterModel BLOCK branch must replace `getModifiedResponse() as GenerateContentResponse` plus `AgentExecutionBlockedError(reason, syntheticResponse, ...)` with “a neutral `ModelOutput` from `hookWireAdapter.afterModelBlockingToModelOutput(...)` carried by `AgentExecutionBlockedError`” (`07-streamprocessor-impl.md:77`). This makes P07 internally contradictory: it requires no `as GenerateContentResponse` and no synthetic response in the streaming AfterModel branch, but the only error transport it names is still typed to `GenerateContentResponse`. A mechanical implementer either breaks typecheck in P07, cheats with a type assertion, or leaves the synthetic response in place.

**Evidence:**
- P07 requires a neutral `ModelOutput` to be carried by `AgentExecutionBlockedError`: `project-plans/issue2349/plan/07-streamprocessor-impl.md:77`.
- Current `AgentExecutionBlockedError` constructor takes `syntheticResponse?: GenerateContentResponse`: `packages/agents/src/core/chatSession.ts:96-105`.

- The current streaming hook path really does cast to `GenerateContentResponse` and throws `AgentExecutionBlockedError` with a synthetic response: `packages/agents/src/core/StreamProcessor.ts:711-720`.

**Required fix:** Make the blocking-error transport neutral before or inside P07. The clean options are: (a) add a small P07 task to retype `AgentExecutionBlockedError` to carry a neutral blocked `ModelOutput`/neutral blocking payload while preserving observable reason/system-message semantics, and update all catch sites; or (b) keep the P07 blocking branch on the old synthetic path and explicitly defer only that branch to P13, but then P07 must not claim the streaming AfterModel hook is fully neutral and P06/P07 acceptance must be adjusted. It does not explicitly require an AST-context-specific allow-list entry for `hookWireAdapter.ts`'s wire-response readers.

Given the overview's rule that bounded exceptions are granted only by a central allow-list with exact file and AST-context (`overview.md:762-783`), leaving the new hook response adapter out of P31's explicit target-state allow-list is a gate-executability risk: either the gate fails on the intended adapter, or an implementer weakens checkF broadly to pass.

## Additional risks / open questions

1. **Hook wire boundary scope remains delicate.** The plan deliberately preserves `HookGenerateContentResponse` as an external JSON wire while banning agents-internal Google shape. That is reasonable, but the boundary must be named and allow-listed exactly. P07/P13/P31 should all converge on the same `hookWireAdapter.ts` AST-context rule; otherwise the gate will either fail intended wire code or accidentally bless a new structural bypass.

2. **Mutation gate may be operationally heavy but is directionally correct.** P33's mutation surface is appropriately broad and avoids the api-only default, but it will be fragile unless per-slice reports are produced consistently. Keep the Stryker triage rule; do not weaken thresholds or exclude migrated files.

3. **Characterization strategy is mostly sound.** The plan correctly distinguishes characterization tests that pass against current behavior from post-flip type-surface tests, especially in P12/P13 and P20/P21. Preserve that separation. Do not let characterization tests assert `{candidates}` or `.parts` internals except in explicit boundary fixtures.

4. **The overview inventory claims checked here are broadly accurate.** Independent checks confirm 46 production `@google/genai` importers under `packages/agents/src`, the seven `ContentConverters.toGeminiContents` production sites, 23 CLI `Contract*` consumers, and 5 non-contract core consumers. These match the overview's appendices and the plan's baseline assumptions.

## Summary

This plan is architecturally aligned with the issue and far better than the rejected #2424 source-swap approach, but it is not yet safe for literal phase-by-phase execution. Fix the P02/P29 enforcement contradictions, make P07's blocking-hook neutralization type-correct, remove the non-existent helper call or add the helper in the neutral-gap phase, replace placeholder pseudocode ranges, and turn diagnostic verification comments into hard failing checks. After those changes, the plan should be re-reviewed against the same criteria before execution.
