# Review of `project-plans/issue2349/overview.md`

## Verdict: APPROVE-WITH-FIXES

The overview is substantially better than a name-based inventory: it correctly identifies the self-inflicted neutral‚ÜíGoogle‚Üíneutral round-trip, the `clientContract.ts` structural-alias problem, the #2424 bridge/alias failure mode, and the need for a structural enforcement gate. I independently verified the major measured counts that anchor the document: 46 production `@google/genai` importers and 54 test/helper importers under `packages/agents/src`, 7 production `ContentConverters.toGeminiContents` call sites, 23 production CLI `Contract*` consumers, 5 production core `Contract*` consumers, and the neutral provider call returning `AsyncIterableIterator<IContent>`.

However, the document is not yet a complete or fully accurate architecture map. It misses several production `.parts`/candidate-content structural dependency sites, overstates the current API event adapter's neutral‚ÜíGemini usage-metadata bridge, and leaves parts of the enforcement-gate story under-specified or internally inconsistent around public-event usage metadata. These are fixable, but they matter: the missing sites are exactly the kind of structural Google-shape residue that caused #2424 to fail.

**Problem:** The document claims to inventory the structural payload access/mutation surface and summarizes it as 6 `.parts`/`candidate.content` readers/mutators, 4 AFC/content-length filters, and 4 usage-key sites.   - The doc lists `ConversationManager.ts:272-277`, `:306`, and `:310`, but not `:34-40` / `:330-345`.
- `clientLlmUtilities.ts` has two production `.parts` readers in the stateless helper path:
  - `packages/agents/src/core/clientLlmUtilities.ts:61-70` extracts prompt text from `Content[]` via `c.parts`.
  - `packages/agents/src/core/clientLlmUtilities.ts:84-92` checks `c.parts?.some(...)` for `next_speaker` fallback behavior.
  - The doc mentions `clientLlmUtilities.ts:143` as `toIContents`, but not these `.parts` reads.
- `streamResponseHelpers.ts` reads candidate content parts during accumulation:
  - `packages/agents/src/core/streamResponseHelpers.ts:101-110` reads `chunk.candidates?.[0]?.content?.parts ?? []` before hook-restricted filtering.
  - The doc lists usage-key reads and `{ role:'model', parts }` history construction, but not this candidate-content read.
- `DirectMessageProcessor.ts` has an additional response-text extractor candidate-content read:
  - `packages/agents/src/core/DirectMessageProcessor.ts:894-899` reads `response.candidates?.[0]?.content?.parts ?? []` through `getResponseTextFromParts(...)`.
  - The doc lists `_ensureResponseText` at `:859-869` and AFC filters, but not `_extractResponseText` at `:894-899`.

**Required fix:** Expand ¬ß2A.4-II and the measured-count summary to include every production `.parts` / `candidate.content` read and mutation, including the four sites above. For each, give a neutral target disposition (`ContentBlock[]` text extraction/consolidation, neutral stateless-helper input handling, neutral chunk accumulation, direct-path `ModelOutput` text extraction). Update ¬ß8 check (f)/(h) and ¬ß9.1-2 so the acceptance criteria cannot pass while these structural dependencies remain.

### Critical 2 ‚Äî The API event-adapter ‚Äúneutral‚ÜíGemini-named bridge‚Äù claim is factually wrong

**Location in doc:** ¬ß7A, especially lines 656-666; also ¬ß2B.1 line 317 and ¬ß7 line 644.

**Problem:** The document says an existing seam ‚Äúalready bridges neutral‚ÜíGemini-named at this boundary‚Äù and specifically claims `eventAdapter.ts` consumes the core-neutral `Finished` event and emits the API `FinishedValue` with Gemini-named `usageMetadata`. That is not what the code does. `eventAdapter.ts` simply casts/stores the `Finished` value and forwards it into `done.finished`; it does not map `UsageStats` fields (`promptTokens`, `completionTokens`, etc.) into `UsageMetadataValue` fields (`promptTokenCount`, `candidatesTokenCount`, etc.). This is both a factual error and an architectural risk: option C is not ‚Äúalready existing‚Äù for `FinishedValue`; it must be implemented if chosen.

**Evidence:**

- Core `Finished` carries neutral `UsageStats`:
  - `packages/core/src/core/turn.ts:241-252` defines `ServerFinishedEvent.value.usageMetadata?: UsageStats`.
  - `packages/agents/src/core/turn.ts:399-406` emits `usageMetadata: chunk.usage` into `AgentEventType.Finished`, where `chunk.usage` is neutral `UsageStats`.
- API `FinishedValue` is declared with Gemini-named usage fields:
  - `packages/agents/src/api/event-types.ts:32-41` defines `UsageMetadataValue` as `promptTokenCount`, `candidatesTokenCount`, `totalTokenCount`, `cachedContentTokenCount`, and `FinishedValue.usageMetadata?: UsageMetadataValue`.
  - `packages/agents/src/api/event-schema.ts:30-39` validates the same Gemini-named shape.
- But `eventAdapter.ts` performs no conversion on `Finished` usage:
  - `packages/agents/src/api/eventAdapter.ts:317-323` casts `e.value as { reason: string; stopReason?: string }`, assigns `state.lastFinished = v`, and yields `makeDone(...)`.
  - `packages/agents/src/api/eventAdapter.ts:229-235` includes `state.lastFinished` verbatim as `done.finished`.
  - The only direct usage event forwarding is also verbatim: `packages/agents/src/api/eventAdapter.ts:267-270` yields `{ type: 'usage', usage: e.value }` for `AgentEventType.UsageMetadata`.

**Required fix:** Rewrite ¬ß7A to distinguish three facts:

1. `ServerFinishedEvent.value.usageMetadata` is neutral.
2. `FinishedValue.usageMetadata` is declared/validated as Gemini-named.
3. The current adapter does **not** bridge between them; it forwards the core finished value verbatim, making the current API typing/schema inconsistent with the emitted runtime value if usage is present.

Then update the options: option C is a required new adapter mapping, not an existing seam. Also add a required planner decision/test for whether `done.finished.usageMetadata` currently contains neutral keys or Gemini-named keys in real emitted API events, because the current code and declared public type disagree.

**Location in doc:** ¬ß8 lines 695-697; ¬ß2A.2 lines 183-191.

**Problem:** The document says the gate should match `ContentConverters.toGeminiContents(` call expressions and allow only the single before-model hook adapter at `streamRequestHelpers.ts:228` if OQ-1a preserves the hook wire. The current line is inside an object literal passed into `applyLLMRequestModifications`; a line-based allow-list alone is brittle because the same function also has a telemetry offender at `streamRequestHelpers.ts:281`. The gate requirement needs to require context validation: the allowed call must be the hook-wire `target.contents` construction immediately converted back via `toIContents`, not just ‚Äúsome call in `streamRequestHelpers.ts`‚Äù.

**Evidence:**

- The legitimate-if-retained hook adapter call is `packages/agents/src/core/streamRequestHelpers.ts:228`.

**Required fix:** Specify that the allow-list for `toGeminiContents` is not merely file-based or comment-based. ### Major 2 ‚Äî Public usage-metadata gate lacks a precise allowed/banned module map and will either miss core-owned events or overreach into public API schema

**Location in doc:** ¬ß7A lines 646-666; ¬ß8 line 697; ¬ß9.1-2b line 722.

## Additional open questions

1. **What is the actual runtime API shape of `done.finished.usageMetadata` today?** The core event emits neutral `UsageStats`, but API types/schema declare Gemini-named `UsageMetadataValue`, and the adapter forwards the value verbatim. Before deciding ¬ß7A option A/B/C, add a characterization test or inspect existing tests to establish whether consumers currently see `promptTokens` or `promptTokenCount` on `done.finished.usageMetadata`.

2. **Should stateless helper paths (`generateJson` / `generateContent`) remain part of the #2349 agent-loop migration or be treated as a separate public/helper contract?** `clientLlmUtilities.ts` and `baseLlmClient.ts` are not only type consumers; they perform text extraction and fallback behavior over `Content.parts`. The overview should explicitly say whether their compatibility surface migrates with `AgentClientContract` or gets a bounded legacy adapter.

3. **What is the telemetry wire contract?** `turnLogging.ts` currently logs Google-shaped request contents and Google-named usage metadata. If telemetry consumers expect those field names, the migration needs a named telemetry serialization boundary. If not, telemetry should become neutral `IContent[]`/`UsageStats` and any Gemini-named telemetry keys should be banned.

4. **Does a future enforcement gate need a source-of-truth allow-list file?** The overview calls for inline `gate-exempt` comments, but a parser-based structural gate with hook-wire/public-wire exceptions may be easier to audit if it also has a central allow-list with exact file, AST pattern, and justification. This would reduce the chance of structural exemptions spreading silently.

5. **Are block-level provider metadata and response-level provider metadata both in scope?** The overview discusses `ModelOutput.providerMetadata` and AFC but should explicitly decide whether provider metadata on individual `ContentBlock`s is preserved, stripped, or left provider/core-only.
