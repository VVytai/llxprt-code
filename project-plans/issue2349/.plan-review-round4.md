# Technical Review: PLAN-20260707-AGENTNEUTRAL

## Verdict: APPROVE-WITH-FIXES

The plan is directionally aligned with issue #2349 and the overview: it rejects the #2424 source-swap failure, names structural Google shape as the real target, plans deletion of the synthetic `GenerateContentResponse` round-trip and both side-channels, covers the `clientContract.ts`/CLI/core blast radius, and adds a structural AST gate with a central allow-list. I spot-checked the key inventory against the current tree: the provider contract is neutral (`RuntimeProvider.ts:77-84`, `RuntimeProviderChat.ts:50`), the current `clientContract.ts` is still Google-shaped (`clientContract.ts:63-123`, `:127-188`), there are currently 46 production `@google/genai` importers in `packages/agents/src`, and there are exactly 7 `toGeminiContents` production sites matching G1-G7. However, the plan still has execution and compliance defects that must be fixed before a coordinator can run it mechanically: early phases are malformed relative to the method docs, several implementation phases do not include required `@pseudocode` markers with concrete line ranges, at least one phase claims full structural-inventory coverage while only checking a subset, and a tooling phase has non-executable package/marker/dependency-update instructions.

## Critical findings

### Critical 1 — Phase 0.5 and 0.6 are executable phases but do not follow the mandatory phase template

**Location in plan:** `project-plans/issue2349/plan/00a-preflight-verification.md:1-149`; `project-plans/issue2349/plan/00b-analysis-pseudocode-verification.md:1-93`; `project-plans/issue2349/plan/execution-tracker.md:7-18`

**Problem:** The plan treats P0.5 and P0.6 as real sequential phases, but they omit mandatory template sections. P0.5 has no `## Requirements Implemented (Expanded)`, no `## Implementation Tasks`, no `## Verification Commands`, no `## Success Criteria`, and no `## Failure Recovery`. P0.6 similarly lacks the required expanded requirements / implementation tasks / verification commands headings. This violates the governing method docs and makes automation/verification inconsistent across the first two gates.

**Evidence:**
- The template says every phase **MUST** include `## Phase ID`, `## Prerequisites`, `## Requirements Implemented (Expanded)`, `## Implementation Tasks`, verification, success, failure recovery, and marker content (`dev-docs/PLAN-TEMPLATE.md:25-76`). It also requires full requirement text and GIVEN/WHEN/THEN behavior (`dev-docs/PLAN-TEMPLATE.md:41-53`).
- P0.5 is a real phase with Phase ID `PLAN-20260707-AGENTNEUTRAL.P0.5` (`plan/00a-preflight-verification.md:1-4`) and the execution tracker makes P0.5/P0.6 part of the contiguous sequence (`plan/execution-tracker.md:7-18`).
- P0.5 jumps from assumptions to blocking issues and a verification gate (`plan/00a-preflight-verification.md:12-149`) without the required phase sections.
- P0.6 is likewise a real phase (`plan/00b-analysis-pseudocode-verification.md:1-18`) but uses ad hoc headings instead of the required `Requirements Implemented`, `Implementation Tasks`, and `Verification Commands` sections (`plan/00b-analysis-pseudocode-verification.md:25-93`).

**Required fix:** Convert P0.5 and P0.6 to full template-compliant phases, including expanded requirements with GIVEN/WHEN/THEN, explicit implementation/verification tasks, success criteria, failure recovery, and completion marker instructions. If the author wants them to be non-template preparatory checklists instead, remove them from the executable phase sequence and execution tracker; but then the method docs’ mandatory Phase 0.5 still needs a compliant replacement.

### Critical 2 — Pseudocode line-marker enforcement is incomplete and self-contradictory

**Location in plan:** `plan/00b-analysis-pseudocode-verification.md:68-72`; `plan/13-directmessageprocessor-impl.md:60-84`; `plan/21-clientcontract-impl.md:32-34`; `plan/verification-template.md:7-14`

**Problem:** The plan says implementation phases must cite pseudocode with `@pseudocode` line markers, but multiple implementation phases only mention pseudocode in prose or use bare “lines X-Y” bullets. A fresh executor following required code-marker rules could complete those phases without adding `@pseudocode lines X-Y` markers to touched functions/classes/tests. This violates PLAN.md’s anti-fraud pseudocode discipline and weakens reviewability.

**Evidence:**
- The shared verification template says implementation phases **MUST** cite `@pseudocode` line numbers (`plan/verification-template.md:7-14`).
- P0.6 tries to enforce this by grepping only a fixed set of phase files for `@pseudocode` (`plan/00b-analysis-pseudocode-verification.md:68-72`), but several mapped implementation phases in that set do not contain actual `@pseudocode` markers.
- P13 names line ranges (`lines 10-19`, `lines 20-22`, etc.) but the required markers at lines 67 and 84 include only `@plan`/`@requirement`, not `@pseudocode lines ...` (`plan/13-directmessageprocessor-impl.md:60-84`).
- P21 likewise says to delete/retype per pseudocode lines 10-41, then lists markers only as `@plan` and `@requirement` (`plan/21-clientcontract-impl.md:32-34`).
- The method doc requires every function/test/class marker to include pseudocode line numbers when applicable (`dev-docs/PLAN-TEMPLATE.md:70-80`) and PLAN.md says implementation phases must explicitly reference numbered pseudocode (`dev-docs/PLAN.md:76-80`).

**Required fix:** For every implementation phase that depends on pseudocode (at least P05, P07, P08, P09, P11, P13, P15, P17, P19, P21, P31, and any other mapped phase), add explicit `@pseudocode lines X-Y` requirements to the per-function/class/test code markers, not only prose bullets. Update P0.6 to mechanically check all mapped implementation phases from the execution tracker, not a hard-coded subset that can drift.

### Critical 3 — P15 claims full §2A.4-II structural-access coverage but verifies only a subset of the overview inventory

**Location in plan:** `plan/15-structural-access-impl.md:81-108`; authoritative inventory `overview.md:274-319`

**Problem:** P15 says it verifies the “FULL overview §2A.4-II inventory” by requirement, but its greps only cover a subset of the listed files. It omits several overview-listed structural access/mutation sites that either must already be proven gone by earlier phases or explicitly checked here: `TurnProcessor._recordOutputContent`, AFC content-length filters in `TurnProcessor`/`DirectMessageProcessor`/`hookToolRestrictions`, `hookToolRestrictions.filterHookRestrictedContent`, and internal usage-key sites in `TurnProcessor`/`turnLogging`. The phase can therefore pass while a known §2A.4-II site remains, unless a later global gate catches it. That is not acceptable for a phase that claims full inventory closure.

**Evidence:**
- The overview lists `streamResponseHelpers.ts:101-108`, `TurnProcessor.ts:798-803`, `hookToolRestrictions.ts:184-192`, and `streamChunkWrapper.ts:77-83` as `.parts` / `candidate.content` readers/mutators (`overview.md:294-297`).
- It lists AFC/content-length filters in `TurnProcessor.ts:728`, `DirectMessageProcessor.ts:386`/`:764`, and `hookToolRestrictions.ts:133` (`overview.md:299-305`).
- It lists internal Google usage-key sites in `TurnProcessor.ts:844-850`, `streamResponseHelpers.ts:149-151`/`:308-314`, `MessageConverter.ts:651-662`, and `turnLogging.ts:85-104` (`overview.md:307-319`).
- P15’s verification greps cover only `ConversationManager.ts`, `clientLlmUtilities.ts`, `streamResponseHelpers.ts`, `MessageStreamOrchestrator.ts`, `MessageStreamTerminalHandler.ts`, `clientHelpers.ts`, `client.ts`, and selected `GeminiContent*` imports (`plan/15-structural-access-impl.md:85-99`). It does not check `TurnProcessor.ts`, `DirectMessageProcessor.ts`, `hookToolRestrictions.ts`, `turnLogging.ts`, or the AFC filters listed above.
- P15 only requires the AST structural count to “strictly decrease” (`plan/15-structural-access-impl.md:101-102`), not to reach zero or to prove all P15-owned inventory sites are gone.

**Required fix:** Either (a) expand P15 verification to include every §2A.4-II site it claims to own, with explicit cross-phase evidence for sites already retired in P08/P11/P13/P19, or (b) narrow P15’s claim to the exact files it owns and add a separate inventory-closure phase that verifies all §2A.4-I/II construction, access, AFC, and usage-key sites are gone or centrally allow-listed. Do not let a “strictly lower” count substitute for closure of a named inventory.

## Major findings

### Major 1 — Core mutation-tooling phase has non-executable dependency-update and marker instructions

**Location in plan:** `plan/01-core-mutation-tooling.md:40-48`, `:74-88`, `:96-111`; current packages `packages/core/package.json`, `packages/agents/package.json:67-68`

**Problem:** P01 adds new devDependencies and a new script to `packages/core/package.json`, but the tasks do not instruct updating `package-lock.json` / running the workspace install needed to make `npm ls --prefix packages/core` succeed. The phase also says to record `@plan`/`@requirement` in a package.json “script description,” which is not a valid concept in npm scripts and contradicts the later instruction that package.json stays valid JSON.

**Evidence:**
- P01 creates `packages/core/stryker.conf.json` and says markers should be included “in the Phase Completion Marker AND in `packages/core/package.json`'s `test:mutation` script description” (`plan/01-core-mutation-tooling.md:40-44`). npm scripts are strings; there is no script-description field.
- P01 modifies only `packages/core/package.json` for devDeps and the `test:mutation` script (`plan/01-core-mutation-tooling.md:74-86`), then repeats that the markers are recorded in the completion marker because JSON tooling files cannot carry source markers (`plan/01-core-mutation-tooling.md:90-92`).
- P01 verification requires `npm ls @stryker-mutator/core --prefix packages/core` and `npm ls @stryker-mutator/vitest-runner --prefix packages/core` to resolve (`plan/01-core-mutation-tooling.md:103-105`), but the implementation tasks never mention updating the lockfile or installing workspace dependencies.
- Current evidence: `packages/agents/package.json` has the Stryker deps (`packages/agents/package.json:67-68`), while `packages/core/package.json` currently has no `stryker`/`test:mutation` entries (verified by grep; no matches in `packages/core/package.json`).

**Required fix:** Add `package-lock.json` to P01’s files-to-modify and specify the exact dependency update command (for example the project’s standard npm workspace install/package-lock update command). Remove the impossible “script description” marker instruction; keep markers in the completion marker or a real Markdown planning artifact only.

### Major 3 — P15/P21 sequencing around `client.getHistory` and `ConversationManager.getHistory` is ambiguous across the contract flip

**Location in plan:** `plan/15-structural-access-impl.md:52-57`, `:70-72`; `plan/21-clientcontract-impl.md:32-38`; current code `packages/agents/src/core/client.ts:403-421`, `packages/agents/src/core/ConversationManager.ts:412-419`

**Problem:** P15 says to delete `toGeminiContents` G1/G2 and make `getHistory` return cloned neutral `IContent[]`, but P21 is the phase that deletes/retypes the cross-package `AgentClientContract`/`AgentChatContract` surface. In the current code, `AgentClient.getHistory()` and `ConversationManager.getHistory()` are still typed/used as Google `Content[]`. If P15 actually changes return types before P21, the monorepo may break unless all dependent callers are updated early; if it only changes internals but keeps the old surface, G1/G2 cannot fully vanish. The plan does not make this boundary compile-green enough.

**Evidence:**
- Current `AgentClient.getHistory()` returns `Promise<Content[]>`; when no chat is live it returns `ContentConverters.toGeminiContents(storedHistory.getAll())` (`packages/agents/src/core/client.ts:403-421`).
- Current `ConversationManager.getHistory()` returns `Content[]` and converts with `ContentConverters.toGeminiContents(iContents)` (`packages/agents/src/core/ConversationManager.ts:412-419`).
- P15 instructs `ConversationManager.getHistory` to return cloned `IContent[]` and delete G2 (`plan/15-structural-access-impl.md:52-57`) and instructs `client.ts` to delete G1 (`plan/15-structural-access-impl.md:70-72`).
- P21 is the phase that deletes payload contract types and retypes `AgentClientContract`/`AgentChatContract` to neutral (`plan/21-clientcontract-impl.md:32-38`).

**Required fix:** Make the compile-green staging explicit. Either move G1/G2 deletion and the `getHistory` return-type flip into P21 with the rest of the contract-surface migration, or expand P15 to include every caller/signature required for `IContent[]` history to compile before P21. Do not leave a phase boundary where `client.ts`/`ConversationManager.ts` are neutral but the public contract still requires `Content[]`.

### Major 4 — The final structural proof depends on P31/P33, but intermediate phases can still “pass” with known structural bypasses

**Location in plan:** `plan/02-ast-gate-skeleton.md:72-84`, `plan/23-subagent-retypes-impl.md:37-41`, `plan/25-executor-retypes-impl.md:39-41`, `plan/27-remaining-retypes-impl.md:49-51`, `plan/33-full-verification.md:37-41`, `:92-101`

**Problem:** The plan has the right final idea: structural-bypass absence is proven by an AST gate, not by grep. But several migration phases rely only on “strictly lower” AST counts or optional advisory greps. That is acceptable for a ratchet only if every phase also identifies its owned structural sites and proves those sites are closed. As noted in Critical 3, that site-level proof is incomplete in at least P15. Otherwise a phase can pass by reducing unrelated count while leaving its own source-swap-like structural path intact.

**Evidence:**
- P02 implements a count mode and baseline, not full fail-mode enforcement (`plan/02-ast-gate-skeleton.md:72-84`).
- Later slice phases use `scripts/agents-neutral-gate.ts --count` as a shrink-ratchet (`plan/23-subagent-retypes-impl.md:37-41`, `plan/25-executor-retypes-impl.md:39-41`, `plan/27-remaining-retypes-impl.md:49-51`).
- P33 correctly says the AST gate is the sole proof for final structural-bypass absence (`plan/33-full-verification.md:37-41`, `:92-101`), but that only protects the final state, not phase-local truth claims.
- The overview warns that the gate must catch raw-import-free structural `.parts` mutation such as `executor-prompt-builder.ts:47-58` (`overview.md:769-778`). A count-only ratchet does not prove a specific file is fixed; it only proves some net count movement.

**Required fix:** For every slice phase, add phase-owned structural-hit identities from the P02/P0.5 baseline and require those exact hit IDs/AST contexts to disappear, not merely a net lower count. Keep the final P31/P33 zero/fail-mode gate, but make per-slice gates site-specific.

### Major 5 — Verification greps for TDD anti-patterns are too brittle and can block valid behavioral tests or miss invalid ones

**Location in plan:** `plan/verification-template.md:29-34`, `:48-66`; `dev-docs/RULES.md:85-108`

**Problem:** The plan correctly mandates no mock theater, no reverse testing, and no structure-only tests. But the shared verification uses broad textual greps such as `toHaveProperty|toBeDefined` and a property-test ratio count based on `it(`/`test(` vs `fc.assert|fc.property|test.prop`. These greps are not semantic: they can fail valid behavioral tests that use `toBeDefined` as part of a value assertion, or pass tests that include a property-based helper but still assert the wrong behavior. The project docs require semantic behavior verification, not only marker/string checks.

**Evidence:**
- RULES.md says tests should cover public API behavior, input→output transformations, edge cases, integration, and must not test implementation details or mock interactions (`dev-docs/RULES.md:85-108`).
- The template’s structure-only detector is a grep over `toHaveProperty|toBeDefined` with a text exclusion for “with specific value” (`plan/verification-template.md:29-34`).
- The property ratio helper counts textual patterns across files (`plan/verification-template.md:48-66`), which is useful as a coarse gate but not enough to prove tests are behavioral.

**Required fix:** Keep the grep as an advisory early warning, but require the verifier to read each flagged assertion and classify whether it is structure-only or part of a behavioral value assertion. For property-based tests, require a short semantic note per property explaining the invariant and why an empty/stub implementation would fail. Do not let a broad grep be the pass/fail authority without human/AST context.

### Minor 2 — P33’s final mutation command is strong but likely very large; the plan needs a failure triage rule for non-actionable Stryker runtime failures

**Location in plan:** `plan/33-full-verification.md:69-90`, `:107-115`

**Problem:** P33 now covers the real migrated surface with archived slice reports plus whole-surface Stryker runs, which is the right intent. But the whole-surface command mutates many large files and all `src/api/**`; if Stryker fails due to runner/runtime/tooling issues rather than surviving mutants, the failure recovery only says route back to owning phases. That may be insufficient for a coordinator executing mechanically.

**Evidence:**
- P33 runs agents Stryker over `StreamProcessor`, `TurnProcessor`, `turn`, `MessageConverter`, `DirectMessageProcessor`, `ConversationManager`, `hookToolRestrictions`, `streamResponseHelpers`, `beforeModelHookDecision`, `src/subagent/**/*.ts`, `src/agents/executor*.ts`, and `src/api/**/*.ts` (`plan/33-full-verification.md:69-83`), plus core llm-types (`plan/33-full-verification.md:86-90`).
- Success criteria require ≥80% for slice reports and whole-surface runs (`plan/33-full-verification.md:107-115`).

**Required fix:** Add a triage step: distinguish (1) mutation score <80, (2) test runner crash/config error, (3) timeout/resource failure, and (4) no-mutants/no-coverage misconfiguration. Require rerunning the smallest failing `--mutate` subset before changing code or weakening the gate.

## Additional risks / open questions

1. **P21 is still the largest atomic risk.** The plan correctly explains why the contract flip is atomic (`plan/21-clientcontract-impl.md:79-80`), but this is a major blast-radius phase touching agents + 23 CLI + 5 core files. It needs the exact path list and a checkpointed diff review before implementation begins.
2. **Hook wire compatibility is intentionally retained for G3.** This is consistent with the domain-model resolution, but the allow-list entry must be AST-context-specific: `streamRequestHelpers.ts` currently has both the allowed hook-wire `toGeminiContents` site at line 228 and a telemetry offender at line 281 (verified by current grep output). A file-level allow-list would recreate the #2424 structural bypass risk.
3. **The core-owned `ServerUsageMetadataEvent` remains outside the agents gate.** The plan documents this limitation, but P19/P33 must paste evidence that the separate core shape check or tracked decision was actually executed; otherwise acceptance §9.1-2b can be overstated.
4. **Current code evidence supports the plan’s main premise.** The provider call is already neutral (`packages/core/src/runtime/contracts/RuntimeProvider.ts:77-84`, `RuntimeProviderChat.ts:50`), while agents still fabricates Google responses (`packages/agents/src/core/MessageConverter.ts:518-540`) and the neutral `toModelStreamChunk` currently does not copy `metadata.providerMetadata` (`packages/core/src/llm-types/modelEnvelope.ts:188-210`). The plan should keep those as hard preflight checks.
