# Technical Review: PLAN-20260707-AGENTNEUTRAL

## Verdict: REJECT

The plan is much stronger than a source-swap plan in intent, but it is not mechanically executable as written. Several phase boundaries would leave the tree uncompilable or leave live Google-shaped hook/stream paths unresolved while claiming neutralization/deletion. The most serious blockers are: P07 tries to remove the streaming synthetic-response conversion without specifying how the still-live AfterModel hook path (which requires `GenerateContentResponse`) is neutralized; P08 deletes `streamChunkWrapper.ts` while other production files still import `chunkToParts`; P13 claims all synthetic response machinery is gone but misses the hook-output/translator contract in core and multiple AfterModel `getModifiedResponse` paths; and many phase files do not meet the method-doc requirement for full GIVEN/WHEN/THEN expansion. These issues would cause either red builds mid-plan or a migration that can still preserve #2424-style structural bypasses.

## Critical findings

### Critical 1 — P07 cannot remove the provider-output synthetic response while the streaming AfterModel hook path still requires `GenerateContentResponse`

**Location in plan:** `plan/07-streamprocessor-impl.md:52-60`, `plan/07-streamprocessor-impl.md:93-117`

**Problem:** P07 says `_convertIContentStream` should use `toModelStreamChunk(iContent)` and delete the provider-output `convertIContentToResponse` call, while preserving behavior. But the actual streaming AfterModel hook implementation is structurally built around a `GenerateContentResponse` converted from the same `IContent` chunk. The plan does not specify a neutral AfterModel hook adapter for streaming at P07, nor does it defer that path explicitly to a later phase. If the implementer follows P07 literally, the code either will not compile or will keep fabricating a synthetic `GenerateContentResponse` for hooks, violating the phase’s claimed neutral streaming pipeline.

**Evidence:**
- P07 requires deleting the provider-output conversion: `plan/07-streamprocessor-impl.md:54-60` says `_convertIContentStream` must use `toModelStreamChunk(iContent)` and delete the `convertIContentToResponse` provider-output call.
- Actual code: `StreamProcessor._convertIContentStream` currently converts each `IContent` into a synthetic response, passes that response into `_processAfterModelHook`, and yields `GenerateContentResponse`: `packages/agents/src/core/StreamProcessor.ts:618-656`.
- Actual code: `_processAfterModelHook` is typed on `GenerateContentResponse`, rebuilds Google-shaped `{ role:'model', parts: [] }`, calls `ContentConverters.toIContent(filteredContent)`, and consumes hook modified responses as `GenerateContentResponse`: `packages/agents/src/core/StreamProcessor.ts:662-736`.
- P07’s build-order caveat only discusses the before-model blocking hook synthetic path, not the streaming AfterModel path: `plan/07-streamprocessor-impl.md:41-50`.

**Required fix:** Split or rewrite P07 so it explicitly neutralizes the streaming AfterModel hook path before/while deleting `convertIContentToResponse` from `_convertIContentStream`. The fix must define the exact neutral hook boundary: how `fireAfterModelEvent` receives byte-compatible hook JSON, how `getModifiedResponse()` / blocking responses are converted into `ModelStreamChunk` or `ModelOutput`, and where the single allowed hook-wire Gemini adapter lives. Add behavioral golden tests for after-model hook modification/blocking on the streaming path before the implementation phase.

### Critical 3 — P13 claims final synthetic-response deletion but does not cover the core hook output contract and all AfterModel modified-response flows

**Location in plan:** `plan/13-directmessageprocessor-impl.md:13-30`, `plan/13-directmessageprocessor-impl.md:95-103`, `plan/13-directmessageprocessor-impl.md:122-150`

**Problem:** P13 states that after the phase, `grep -rn "convertIContentToResponse\|patchMissingFinishReason\|getSyntheticResponse" packages/agents/src | grep -v test` is empty and no synthetic response is fabricated anywhere. That check is scoped only to `packages/agents/src`, but the hook API that manufactures hook `llm_response` into a `HookGenerateContentResponse` lives in core. The actual agents direct and streaming paths also consume `getModifiedResponse()` as `GenerateContentResponse`. The plan does not specify retyping core hook output classes/translators or the after-model modified response path in a way that satisfies the governing principle.

**Evidence:**
- Actual core hook output still exposes synthetic responses: `packages/core/src/hooks/types.ts:323-339` defines `BeforeModelHookOutput.getSyntheticResponse()` returning `HookGenerateContentResponse`, using `defaultHookTranslator.fromHookLLMResponse`.
- Actual streaming AfterModel path consumes `getModifiedResponse()` as `GenerateContentResponse`: `packages/agents/src/core/StreamProcessor.ts:711-733`.
- Actual direct AfterModel path consumes `getModifiedResponse()` as `GenerateContentResponse`: `packages/agents/src/core/DirectMessageProcessor.ts:816-846`.
- P13 only names `beforeModelHookDecision.ts`, `StreamProcessor._patchMissingFinishReason`, `streamRequestHelpers.patchMissingFinishReason`, and `providerStopReason.ts` for final blocking-hook deletion: `plan/13-directmessageprocessor-impl.md:95-103`.
- P13’s proof command is agents-scoped and therefore cannot detect the still-Google-shaped core hook output contract: `plan/13-directmessageprocessor-impl.md:128-131`.
- The issue’s governing principle says only Gemini provider/conversion code directly behind it may use Google/Gemini SDK types; shared contracts are wrong if defined in Google-shaped types. The current core hook surface is a shared contract, not Gemini-provider-only.

**Required fix:** Add an explicit phase (or expand P13/P21) to retype the hook response contract in `packages/core/src/hooks/types.ts` and any translator boundary to neutral domain types, while preserving the external hook JSON wire via a single named adapter. Include both before-model and after-model hook modified-response paths. Verification must search core + agents, not only `packages/agents/src`, for `getSyntheticResponse`, `HookGenerateContentResponse`, `llm_response` translator output, and `GenerateContentResponse`-shaped hook response flows.

### Critical 4 — P11 explicitly allows a residual `GenerateContentResponse` helper in `hookToolRestrictions.ts`, contradicting the plan’s own “side-channel retired by deletion” requirement unless tightly time-bounded and mechanically enforced

**Location in plan:** `plan/11-side-channel-retirement-impl.md:46-52`, `plan/11-side-channel-retirement-impl.md:66-90`

**Problem:** The plan title says “Side-channel retirement — IMPL” and the requirement says no WeakMap/Symbol identity keying on `GenerateContentResponse`/`FunctionCall`. But P11 allows keeping a `GenerateContentResponse`-shaped helper for the before-model blocking path, potentially allow-listed. This is a source-swap risk: a coordinator can satisfy P11 while leaving a Google-shaped adapter in the side-channel module. The later P13 says it removes this, but P11’s success criteria allow it and the plan does not include a hard P13 verification for `hookToolRestrictions.ts` specifically.

**Evidence:**
- P11 says to keep a bounded `GenerateContentResponse` path if cleaner: `plan/11-side-channel-retirement-impl.md:50`.
- P11 verification permits `@google/genai` in `hookToolRestrictions.ts` if allow-listed: `plan/11-side-channel-retirement-impl.md:70-74`.
- Actual current `hookToolRestrictions.ts` is the WeakMap/Symbol side-channel module and has many Google-shaped filtering functions: `packages/agents/src/core/hookToolRestrictions.ts:98`, `:130`, `:136`, `:184`, `:191`, `:255`, `:290`, `:295`.
- The specification states “Side-channels are RETIRED by deletion, not adaptation” and says `hookToolRestrictions.ts` WeakMap/Symbol identity keying is replaced by explicit `HookRestrictions`: `specification.md:21`.

**Required fix:** Do not allow a residual `GenerateContentResponse` helper in the side-channel module as a P11 success state. If a temporary compatibility function is unavoidable, place it in a clearly named hook-wire boundary module, give it an owning deletion phase, and add P13 verification that specifically asserts `hookToolRestrictions.ts` has zero `GenerateContentResponse`, zero `@google/genai`, zero WeakMap/Symbol, and no allow-list entry remains for it.

### Critical 5 — P03 writes production stubs before failing tests, weakening the plan’s own TDD guarantee

**Location in plan:** `plan/03-neutral-gap-types-stub.md:47-73`, `plan/04-neutral-gap-types-tdd.md:55-90`

**Problem:** The first neutral gap type phase creates production code stubs (`agentMessageInput.ts`, changes to `modelEnvelope.ts`, barrel exports) before P04 writes the failing tests. This conflicts with the governing TDD rule as stated in `dev-docs/PLAN.md` and `dev-docs/RULES.md`: production code must be written in response to a failing test. A stub-first phase can still be useful in some templates, but this plan is being judged against “TDD implementation PLAN” and explicitly says no production code should precede tests except tooling/provisioning.

**Evidence:**
- PLAN core principle: “Every line of production code must be written in response to a failing test”: `dev-docs/PLAN.md:11`.
- RULES repeats TDD and rejects test-after development: `dev-docs/RULES.md:194-204`.
- P03 creates `packages/core/src/llm-types/agentMessageInput.ts`, modifies `modelEnvelope.ts`, and exports new symbols before any P04 failing tests exist: `plan/03-neutral-gap-types-stub.md:47-60`.
- P04 writes the tests only after P03: `plan/04-neutral-gap-types-tdd.md:55-90`.

**Required fix:** Reorder neutral gap types as TDD-first: P03 should create failing tests against the desired public exports (or use compile-failing type tests if the symbol does not exist), then P04/P05 should add minimal implementation and final implementation. Alternatively, explicitly justify a template-sanctioned stub phase with no behavioral production logic and ensure the verification records that the stubs are uncallable placeholders, but the stricter TDD-compliant fix is test-first.

### Major 4 — Characterization tests sometimes permit reading current Google-shaped surfaces in a way that can accidentally lock in internals

**Location in plan:** `plan/12-direct-message-characterization-tdd.md:16`, `plan/12-direct-message-characterization-tdd.md:34-43`, `plan/20-clientcontract-tdd.md:16`, `plan/20-clientcontract-tdd.md:40-43`

**Problem:** For a migration, characterization through current APIs is reasonable. However, P12 and P20 permit reading current `GenerateContentResponse` / `Contract*` values to capture golden values, and the guard is “do not assert internal structure.” This is too easy to implement incorrectly: a coordinator can read `.text` safely, but “current accessor” is ambiguous because current response objects expose `candidates`, `usageMetadata`, `functionCalls`, etc. The plan needs a strict helper/adapter to extract observable text/usage/history without allowing candidate/parts indexing.

**Evidence:**
- P12 allows reading current `GenerateContentResponse` accessors: `plan/12-direct-message-characterization-tdd.md:16`, `:34-43`.
- P20 allows reading current `Contract*`-typed return values: `plan/20-clientcontract-tdd.md:16`, `:40-43`.
- Actual direct response exposes Google-shaped internals and text getter: `packages/agents/src/core/DirectMessageProcessor.ts:166-169`, `:806-879`, `:894-899`.
- `RULES.md` says not to test implementation details or mock interactions: `dev-docs/RULES.md:102-108`.

**Required fix:** Define phase-local observer helpers that expose only observable values (`visibleText`, public event sequence, committed `HistoryService` state, usage counts mapped to neutral names) and ban direct indexing into `candidates`, `.parts`, `usageMetadata`, or `Contract*` payload internals in characterization tests. Add verification greps/AST checks over the new tests for those forbidden accesses.

### Major 7 — The verification plan relies heavily on grep commands that can miss AST/context issues or be syntactically weak

**Location in plan:** examples in `plan/07-streamprocessor-impl.md:96-112`, `plan/13-directmessageprocessor-impl.md:126-135`, `plan/15-structural-access-impl.md:132-151`, `plan/33-full-verification.md:101-111`

**Problem:** The plan correctly says the AST gate is authoritative, but many phase-level success criteria still depend on grep snippets. Several grep checks are either too narrow (file-local) or too broad (will match comments/types) and can produce false confidence. Example: P13 greps `sendMessage` and `ModelOutput` on the same line, which is not a reliable TypeScript signature check.

**Evidence:**
- P13 uses `grep -n "sendMessage" ... | grep -q "ModelOutput"`: `plan/13-directmessageprocessor-impl.md:134`; a multiline signature or comment could defeat or satisfy this incorrectly.
- P15 permits residual `toGeminiContents` by counting lines only in two files: `plan/15-structural-access-impl.md:148-151`, but does not prove no new call in another file in that phase.
- P33 does have an AST gate and says greps are supplemental: `plan/33-full-verification.md:117-125`, but earlier phase success criteria can still be passed or failed by weaker grep checks before the full gate exists.

**Required fix:** For phase gates, use the P02/P31 AST `--by-file` output for structural assertions and TypeScript typecheck/API tests for signature assertions. Keep greps as human diagnostics only unless they are exact symbol-reference checks backed by AST output.

## Additional risks / open questions

1. **Hook wire compatibility is the highest-risk unresolved seam.** The plan correctly adds hook-wire goldens in P10, but the implementation phases do not comprehensively explain how the public hook JSON wire remains Gemini-shaped while the internal hook response/request types become neutral. The required adapter boundary must be specified for before-model request, before-model blocking response, after-model response modification, and before-tool-selection.

2. **Cross-package `clientContract` flip remains a large atomic phase.** P21 touches core, agents, 23 CLI consumers, and 5 core consumers. The plan acknowledges build-green requirements, but P21 is still a very large blast radius. Consider splitting by compile-preserving compatibility adapters only if the adapters are neutral-domain typed and do not preserve Contract* payload aliases.

3. **Mutation tooling addition in P01 can churn the root lockfile.** The plan requires `npm install`, which may update more than the two intended Stryker dependencies. P01 verification should include a lockfile diff review proving no unrelated dependency family drift occurred.

4. **`ModelOutput.afcHistory` mutates an already-merged core neutral type.** Actual `ModelOutput` exists and currently lacks `afcHistory`: `packages/core/src/llm-types/modelEnvelope.ts:51-59`. The plan adds it, but should verify all core exports and consumers remain semantically neutral and not Gemini-AFC-specific in naming/docs.

5. **Public usage metadata shape is a core-owned boundary.** P33 adds a compensating check for `ServerUsageMetadataEvent`, which is good. Ensure P19 actually creates a core test in `packages/core/src/core/__tests__/serverUsageMetadataEvent.shape.test.ts`; otherwise P33’s command will fail late.

## Summary

This plan has the right architectural intent and a strong inventory, but it is not safe to execute literally. The blocking issues are executable, not cosmetic: P07/P08/P13 have build-order and unhandled-hook-path contradictions that would either break the build or preserve Google-shaped synthetic response paths after the plan claims they are gone. The plan should be rejected until those phase boundaries are corrected, the hook response/request boundary is specified end-to-end, and method-doc/template compliance is tightened across all phases.
