# Technical Review — PLAN-20260707-AGENTNEUTRAL

## Verdict: REJECT

The plan has the right architectural intent in several places (neutral end-to-end stream currency, deletion of `streamChunkWrapper.ts`/`providerStopReason.ts`, AST gate with central allow-list), but it is not executable as written. The blockers are not cosmetic: multiple phases violate the governing plan template, several implementation phases merely reference requirements instead of expanding them, TDD is incomplete for broad migration slices, one phase deletes `MessageConverter.convertIContentToResponse` before a later phase still needs it, several open questions remain as “DEFAULT/if confirmed” decisions, and the final dependency-removal phase explicitly permits a residual agents `@google/genai` dev dependency. A coordinator executing this mechanically would either break the build mid-sequence or produce another source-swap/partial migration that the final gate may not actually prove.

---

## Critical findings

### Critical 1 — Method-doc noncompliance: most phases do not follow the mandatory phase template

**Location in plan:** `project-plans/issue2349/plan/*.md`, especially verification siblings and TDD phases.

**Problem:** `PLAN-TEMPLATE.md` says each phase “MUST follow this structure” and requires `Phase ID`, `Prerequisites`, expanded requirements with full GIVEN/WHEN/THEN text, `Implementation Tasks`, `Verification Commands`, `Success Criteria`, `Failure Recovery`, and `Phase Completion Marker` (`dev-docs/PLAN-TEMPLATE.md:23-83`). The plan omits required sections across many phase files. This is not harmless formatting: the coordinator relies on those sections to decide whether to proceed and how to recover.

**Evidence:**

- The governing template explicitly requires the full phase structure (`dev-docs/PLAN-TEMPLATE.md:23-83`) and says expanded requirements must copy full text, not just reference it (`dev-docs/PLAN-TEMPLATE.md:41-53`).
- `04-neutral-gap-types-tdd.md` has no `## Implementation Tasks`, no `## Success Criteria`, and no `## Failure Recovery`; it jumps from test tasks to verification and marker (`project-plans/issue2349/plan/04-neutral-gap-types-tdd.md:17-58`).
- `07-streamprocessor-impl.md` has no `## Failure Recovery` (`project-plans/issue2349/plan/07-streamprocessor-impl.md:31-43`).
- `20-clientcontract-impl.md` has no `## Failure Recovery` (`project-plans/issue2349/plan/20-clientcontract-impl.md:33-44`).
- Verification siblings such as `03a-neutral-gap-types-stub-verification.md` contain checks and a marker but do not contain expanded requirements, implementation tasks, verification commands, success criteria, or failure recovery (`project-plans/issue2349/plan/03a-neutral-gap-types-stub-verification.md:1-26`).

**Required fix:** Normalize every phase file, including NNa verification phases and TDD phases, to the template. For each phase add: complete expanded requirements with GIVEN/WHEN/THEN and Why This Matters; concrete implementation/test tasks; concrete verification commands; success criteria; failure recovery; completion marker contents. Do not rely on `verification-template.md` as a substitute for required per-phase sections.

### Critical 2 — Requirements are not expanded inline; many phases explicitly defer to `specification.md`

**Location in plan:** implementation/TDD phases including P04, P05, P07, P08, P09, P11, P13, P15, P18, P20, P24, P25.

**Problem:** The template requires complete requirement text in each phase and says “DO NOT just reference” (`dev-docs/PLAN-TEMPLATE.md:41-53`). The plan repeatedly says “Full text in specification.md” or “copy verbatim into code headers,” which defeats the anti-fraud purpose of expanded requirements. A fresh subagent executing one phase can miss acceptance details such as REQ-002.5 public event shape, REQ-001.3 neutral request DTO, or REQ-007.3 core-owned event limitation.

**Evidence:**

- P04 uses a grouped heading for four requirements and then says “Full text in specification.md; copy verbatim into the test-file headers” instead of expanding them in the phase (`project-plans/issue2349/plan/04-neutral-gap-types-tdd.md:10-15`).
- P07 lists requirement IDs and says “Full text in specification.md — copy verbatim into code headers” (`project-plans/issue2349/plan/07-streamprocessor-impl.md:10-11`).
- P20 does the same for the cross-package contract flip (`project-plans/issue2349/plan/20-clientcontract-impl.md:10-11`).
- The requirements being referenced are detailed and materially important; for example REQ-002.1 through REQ-002.5 include no synthetic response, direct chunk wrapping, Turn block consumption, deletion of fabricators, and unchanged public event shape (`project-plans/issue2349/specification.md:97-102`).

**Required fix:** For every phase, copy the full text for each implemented requirement from `specification.md`, split grouped IDs into individual `### REQ-*` sections, and include full GIVEN/WHEN/THEN behavior. Verification siblings must also state what requirement behavior they are verifying, not just “follow verification-template.”

### Critical 3 — Phase ordering is build-breaking: P09 deletes `convertIContentToResponse` before P13 neutralizes the direct path that still uses it

**Location in plan:** P09 vs P13.

**Problem:** P09 instructs deletion of `MessageConverter.convertIContentToResponse` and claims the callers are already neutralized by “P07/P08 + P13,” but P13 is a later phase. The direct-message path still uses `convertIContentToResponse` until P13. Executing P09 before P13 will break typecheck/build or force an unplanned partial implementation in P09.

**Evidence:**

- P09 says to delete `convertIContentToResponse` and then asserts the streaming/direct synthetic paths are “already neutralized P07/P08 + P13” (`project-plans/issue2349/plan/09-messageconverter-impl.md:15-18`). P13 is later in the sequence (`project-plans/issue2349/plan/13-directmessageprocessor-impl.md:1-15`).
- P13 itself says it will delete the direct path’s after-model `convertIContentToResponse` fabrication (`project-plans/issue2349/plan/13-directmessageprocessor-impl.md:15-22`).
- The overview identifies the direct path fabrication as a distinct site: `_processDirectResponse` builds a `GenerateContentResponse` via `convertIContentToResponse(lastResponse)` (`project-plans/issue2349/overview.md:336-348`).
- The current code still types the non-streaming path as Google-shaped: `TurnProcessor.sendMessage(...)` returns `Promise<GenerateContentResponse>` (`packages/agents/src/core/TurnProcessor.ts:130-133`).

**Required fix:** Reorder the phases or split deletion. Either move P09 after P13, or in P09 only remove streaming callers and leave the function until P13 deletes the direct caller, then add a later “final MessageConverter deletion” phase. The sequence must have a compile-green checkpoint after each phase.

### Critical 4 — P08 changes `TurnProcessor.sendMessage` to `ModelOutput` before `DirectMessageProcessor` is neutralized

**Location in plan:** P08 and P13.

**Problem:** P08 requires `TurnProcessor.sendMessage` to return `Promise<ModelOutput>` and handle AFC/output recording neutrally, while P13 later changes `DirectMessageProcessor.generateDirectMessage` to return `ModelOutput` and deletes the direct-path synthetic response. This ordering is suspect because `sendMessage` currently returns the direct response as `GenerateContentResponse`; P08 would need to do the direct-path conversion early, duplicating or preempting P13.

**Evidence:**

- P08 tasks require `sendMessage: return Promise<ModelOutput>` (`project-plans/issue2349/plan/08-turnprocessor-turn-impl.md:15-17`).
- P13 is the phase that actually neutralizes `DirectMessageProcessor` to `Promise<ModelOutput>` (`project-plans/issue2349/plan/13-directmessageprocessor-impl.md:15-22`).
- Current code shows `sendMessage` returns `Promise<GenerateContentResponse>` and returns the response from `_executeSendWithRetry` (`packages/agents/src/core/TurnProcessor.ts:130-160`).
- The overview says the target direct path returns `ModelOutput`, and that this is part of direct-message neutralization (`project-plans/issue2349/overview.md:336-348`).

**Required fix:** Keep `TurnProcessor.sendMessage` return-type flip in the same phase as the direct path neutralization, or explicitly define a temporary adapter with compile-green proof and delete it in P13. The cleaner fix is to move `sendMessage` direct-return work from P08 to P13.

### Critical 5 — TDD is incomplete for broad production slices; several implementation phases have no preceding failing behavioral tests

**Location in plan:** P16, P21, P26, and parts of P18/P20.

**Problem:** The governing docs state “Every line of production code must be written in response to a failing test” (`dev-docs/RULES.md:3-5`) and “TDD is MANDATORY” (`dev-docs/PLAN.md:9-21`). Several phases are large implementation-only sweeps with no prior RED phase. P16 embeds “Test Tasks” and implementation in one phase, but does not define a separate failing TDD phase or verification that tests fail before implementation. P21 is a massive remaining-retypes phase across subagents/executor/compression/agenticLoop/api/TodoContinuation/chatSession/etc. without its own characterization tests for those behaviors.

**Evidence:**

- TDD mandate: `dev-docs/RULES.md:3-5`, `dev-docs/PLAN.md:11-21`.
- P16 combines test tasks and implementation tasks in the same implementation phase (`project-plans/issue2349/plan/16-runtime-enums-googleparthelpers-impl.md:16-30`) and has no separate P16 TDD phase that must fail before implementation.
- P21 retypes a very broad list of remaining agents files (`project-plans/issue2349/plan/21-remaining-retypes-impl.md:16-31`) but only says “keep existing behavioral tests green” and does not identify slice-specific failing tests.
- P21 covers risky structural sites such as subagent/executor `{role:'user',parts}` constructions and `executor-prompt-builder.ts` generic `.parts` mutation (`project-plans/issue2349/plan/21-remaining-retypes-impl.md:24-27`), while the overview shows these include a raw-import-free structural bypass case (`project-plans/issue2349/overview.md:256-268`).

**Required fix:** Split P16 into TDD and implementation phases. Split P21 into smaller vertical slices, each with a preceding characterization/TDD phase covering observable behavior (subagent run, executor initial messages, tool-response feed, compression/provider-content enforcement, API session control, agenticLoop cancelled-tool history, TodoContinuation, chatSession facade). Each slice must prove tests fail before implementation or explicitly identify existing characterization tests that already fail against the planned neutral API.

### Critical 7 — The plan leaves “resolved” open questions as defaults/conditionals, creating non-deterministic execution

**Location in plan:** `analysis/domain-model.md` OQ resolutions and dependent phases.

**Problem:** The user asked whether open questions are resolved with concrete decisions. Several “resolutions” are still conditional defaults requiring future confirmation. That makes execution non-deterministic and can change gate allow-list, public API shape, telemetry keys, and executor prompt schema mid-run.

**Evidence:**

- The domain model says some genuine choices are handled by a “DEFAULT” and require preflight/characterization to confirm before dependent slices proceed (`project-plans/issue2349/analysis/domain-model.md:109-112`).
- OQ-1a says hook wire stays Gemini-shaped by default, but preflight/characterization may delete G3 instead (`project-plans/issue2349/analysis/domain-model.md:113-114`).
- OQ-2u defaults to option C but may escalate to option B depending on characterization (`project-plans/issue2349/analysis/domain-model.md:119-120`).
- OQ-3t allows a telemetry exception if consumers require Gemini keys (`project-plans/issue2349/analysis/domain-model.md:123`).
- OQ-12 leaves a public schema decision as “migrate to `IContent[]` OR keep legacy `Content[]` converted at ingestion” (`project-plans/issue2349/analysis/domain-model.md:133`).

**Required fix:** Resolve these before implementation. If characterization is needed, add explicit decision gates that update the plan before dependent phases, not ambiguous branches inside the plan. The final plan must say exactly whether G3 survives, whether usage public shape is B or C, whether telemetry has a Gemini-key exception, and whether `PromptConfig.initialMessages` is neutral or legacy-boundary.

### Critical 8 — Dependency removal phase contradicts the issue requirement by permitting a residual agents `@google/genai` dev dependency

**Location in plan:** P26.

**Problem:** The issue requires removing `@google/genai` from `packages/agents/package.json` at the end. P26 permits retaining it as a dev dependency “if strictly required” for allow-listed tests. That violates the stated scope: `packages/agents` must be off `@google/genai`, and tests should use neutral fixtures or a bounded converter fixture that does not require the agents package to depend on the SDK.

**Evidence:**

- Specification says `@google/genai` is present today and “is REMOVED at the end of this plan” (`project-plans/issue2349/specification.md:46-52`).
- REQ-013.1 says after zero imports, `@google/genai` is removed from `packages/agents/package.json` (`project-plans/issue2349/specification.md:140-141`).
- P26 allows “keep the devDependency ONLY if strictly required” (`project-plans/issue2349/plan/26-remove-genai-dependency.md:15-17`) and its verification command allows “documented dev-only” presence (`project-plans/issue2349/plan/26-remove-genai-dependency.md:21-25`).

**Required fix:** Remove the dev-only escape hatch. No `@google/genai` entry may remain in `packages/agents/package.json`. If allow-listed tests need structural Gemini fixtures, express them as plain structural objects typed locally/unknown or move SDK-specific tests to the Gemini provider/conversion package where the SDK is allowed.

### Critical 9 — The plan does not require an actual ≥80% mutation-score gate

**Location in plan:** `verification-template.md`, P27.

**Problem:** The user specifically asked to verify ≥80% mutation-score gates are specified. The plan says “target ≥80% mutation” and permits “spot-check via reasoning if a full stryker run is impractical.” That is not a gate. P27 does not run mutation testing at all.

## Major findings

### Major 1 — P21 is too broad and under-specified to prevent a source-swap or missed structural site

**Location in plan:** P21.

**Problem:** P21 attempts to mop up many remaining files in one phase with generic substitutions. This is precisely the kind of bulk source-swap that #2424 demonstrated is unsafe. The overview’s structural inventory is line-level and disposition-specific, but P21 does not enumerate exact sites or old code removed for each file.

**Evidence:**

- The overview inventories structural construction sites in subagent/executor code, including `executor-prompt-builder.ts:47-58` as a raw-import-free `.parts` mutator (`project-plans/issue2349/overview.md:256-268`).
- The overview also inventories `.parts`/`candidate.content`/usage-key access/mutation sites across many files (`project-plans/issue2349/overview.md:270-319`).
- P21 only lists broad categories and says to retype imports and constructions generically (`project-plans/issue2349/plan/21-remaining-retypes-impl.md:16-31`).

**Required fix:** Replace P21 with multiple file-group phases or a detailed checklist mapping each overview §3.2 / §2A.4 site to an exact modification, deletion, or bounded adapter. Each site must have a verification grep/AST check and behavioral test reference.

### Major 2 — Enforcement gate implementation is not fully tied to package scripts despite claiming CI npm scripts

**Location in plan:** P25.

**Problem:** P25 says to wire gates into CI “as new npm scripts,” but its implementation tasks only mention `.github/workflows/ci.yml` and do not include root/package `package.json` script edits. Without package scripts, CI wiring will either call non-existent scripts or bypass the intended local verification path.

**Evidence:**

- Enforcement pseudocode says the gate wires into CI “as a new npm script” (`project-plans/issue2349/analysis/pseudocode/enforcement-gate.md:87-90`).
- P25 says “Wire both gates into `.github/workflows/ci.yml` as new npm scripts” (`project-plans/issue2349/plan/25-enforcement-gate-impl.md:14-18`) but does not list `package.json` under files to modify.
- P27 runs `npx tsx scripts/agents-neutral-gate.ts` directly rather than `npm run lint:agents-neutral-gate`, so it would not catch missing package scripts (`project-plans/issue2349/plan/27-full-verification.md:15-30`).

**Required fix:** Add root `package.json` script changes to P25 (and verification that `npm run lint:agents-neutral-gate` and `npm run lint:agents-neutral-test-gate` both pass). P27 should run the npm scripts as CI will run them.

### Major 3 — Gate scope intentionally cannot enforce core-owned public usage shape, but the plan does not add a compensating core check

**Location in plan:** OQ-8 / REQ-007.3 / P18 / P25.

**Problem:** The plan says the agents gate cannot enforce `packages/core/src/core/turn.ts` usage metadata shape and that a separate core-package check is out of scope unless preflight elects to add it. But the issue’s governing principle includes “core services/history, the client contract, CLI” being neutral, and the overview explicitly calls out public event usage metadata. Leaving this as documentation only allows a future core regression outside the agents gate.

**Evidence:**

- Enforcement pseudocode repeats that limitation (`project-plans/issue2349/analysis/pseudocode/enforcement-gate.md:79-85`).
- Specification REQ-007.3 requires the scope limitation to be stated and either a separate core check added or tracked in cross-package migration (`project-plans/issue2349/specification.md:121-124`).

**Required fix:** Add a small core/public-event gate or explicit P20/P25 verification that the core-owned event types remain in the chosen shape and cannot silently reintroduce internal Google-shaped usage. Documentation alone is insufficient for a CI-enforced migration.

### Major 4 — The plan’s characterization tests risk passing against adapters rather than proving the synthetic round-trip is gone

**Location in plan:** P06, P10, P12, P14, P19, P22.

**Problem:** The characterization strategy is correct in spirit, but most tests pin observable behavior only. Observable behavior alone cannot prove the internal synthetic-response round-trip is removed; that proof is delegated to later greps/gate. If the gate is delayed until P25, earlier phases P07-P21 can pass behavior while leaving hidden structural bypasses.

**Evidence:**

- P06 explicitly forbids assertions on `GenerateContentResponse`/`{candidates}`/`.parts` and checks only behavior (`project-plans/issue2349/plan/06-stream-pipeline-characterization-tdd.md:27-43`).
- P25, much later, is the first phase that implements the structural AST gate (`project-plans/issue2349/plan/25-enforcement-gate-impl.md:1-26`).
- The overview warns that structural bypasses can be raw-import clean and must be caught by structural checks, not just import checks (`project-plans/issue2349/overview.md:164-191`, `project-plans/issue2349/overview.md:208-319`).

**Required fix:** Add an interim lightweight structural check after each migration slice (or move the gate earlier in stub/TDD form and run it in “expected hits shrink” mode) so P07-P21 cannot preserve behavior with a hidden synthetic/Google-shaped adapter.

## Minor findings

### Minor 1 — Plan markers use both colon and space forms inconsistently

**Location in plan:** many phase files.

**Problem:** Template examples use both `@plan:...` in prose checks and `@plan PLAN...` in code markers. The plan’s verification commands mostly grep colon form (`@plan:PLAN-...`) while required code marker snippets use space form (`@plan PLAN-...`). This can make marker verification falsely fail.

**Evidence:**

- Template code marker uses `@plan PLAN-...` (`dev-docs/PLAN-TEMPLATE.md:70-80`), while template grep uses `@plan:...` (`dev-docs/PLAN-TEMPLATE.md:87-94`).
- P03 required code marker uses space form (`project-plans/issue2349/plan/03-neutral-gap-types-stub.md:40-46`), but P03 verification greps colon form (`project-plans/issue2349/plan/03-neutral-gap-types-stub.md:55-59`).

**Required fix:** Standardize marker syntax for this plan and make all grep commands match it, or grep both forms intentionally.

---

### Minor 2 — Some verification commands use brittle grep patterns that do not match the symbols they intend

**Location in plan:** P20, P27.

**Problem:** Several grep commands use basic regex syntax with escaped parentheses in a way that may not match as intended across platforms. This is a verification reliability issue for a plan that relies heavily on grep.

**Evidence:**

- P20 uses `grep -rn "Contract\(Part\|Content\|...\)"` (`project-plans/issue2349/plan/20-clientcontract-impl.md:33-37`). Basic grep treats `\|` differently than `grep -E`, and this may miss occurrences depending on implementation.
- P27 uses broad greps that allow an exception comment for `toGeminiContents` but do not distinguish G3 from forbidden telemetry sites (`project-plans/issue2349/plan/27-full-verification.md:25-28`).

**Required fix:** Use `grep -E` consistently for alternation or rely on the AST gate for structural checks. For `toGeminiContents`, use the allow-list-aware gate rather than broad grep exceptions.

**Location in plan:** P0.5.

**Problem:** Preflight is supposed to be executed and pasted by the coordinator before implementation. The file already contains “STATUS (spot-verified)” statements, which can encourage a coordinator to skip actual verification.

## Additional risks / open questions

1. **Contract compatibility vs breaking schema remains unresolved.** OQ-12 still gives two incompatible paths for `PromptConfig.initialMessages` (`project-plans/issue2349/analysis/domain-model.md:133`). This is a public/extension-facing decision and should not be left to an executor phase.
2. **Public usage shape may become a breaking change without an explicit compatibility decision.** OQ-2u allows option C or B depending on runtime characterization (`project-plans/issue2349/analysis/domain-model.md:119-120`). The plan must state the final public API commitment before P18.
3. **Phase size risk:** P20 and P21 are too large to execute atomically with high confidence. P20 touches `clientContract.ts`, agents client, 23 CLI consumers, and 5 core consumers in one build-green phase (`project-plans/issue2349/plan/20-clientcontract-impl.md:13-31`). That may be necessary for the surface flip, but it needs a more exhaustive file list and rollback plan.
4. **Gate arrives late:** Because structural enforcement lands at P25, a large amount of migration work can proceed before the real anti-#2424 detector exists. Consider moving gate stub/TDD earlier and running it in shrink-ratchet mode throughout.
5. **Actual code confirms the plan’s core premise, but also confirms the migration is very sensitive to ordering.** Provider output is neutral (`packages/core/src/runtime/contracts/RuntimeProvider.ts:77-84`), while `clientContract.ts` is still deeply Google-shaped (`packages/core/src/core/clientContract.ts:63-125`, `packages/core/src/core/clientContract.ts:127-188`) and `toModelStreamChunk` currently drops response-level provider metadata (`packages/core/src/llm-types/modelEnvelope.ts:188-210`). Those facts support the plan direction but make the ordering defects above critical.
