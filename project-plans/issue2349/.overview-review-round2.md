# Review of `project-plans/issue2349/overview.md`

## Verdict: APPROVE-WITH-FIXES

The overview is substantially better than a name-only import inventory: it correctly identifies the self-inflicted `IContent -> GenerateContentResponse -> ModelStreamChunk -> Part[]` round-trip, the `clientContract.ts` payload-alias problem, the two side-channels, the structural `ContentConverters.toGeminiContents()` bypass, and the cross-package blast radius. I independently verified the headline counts (46 production `@google/genai` importers under `packages/agents/src`, 100 total importers, 7 `toGeminiContents` production call sites, 0 named `GeminiContent*` references in agents production, 23 CLI + 5 core `Contract*` production consumers). However, the document still has several material gaps and one incorrect inventory assertion that would mislead a later implementation plan, especially around anonymous `{role, parts}` structural flows, direct-message/blocking-hook synthetic responses, usage metadata mapping, and exact gate semantics.

## Critical findings

None.

## Major findings

### Major 1 — Anonymous structural Google-envelope inventory is materially incomplete

**Location in doc:** §2A.4 lines 206-209; §8 lines 543-545; §9.1 lines 559-562.

**Problem:** The document claims the one anonymous Gemini-envelope fabricator is `MessageConverter.convertIContentToResponse`, with only two AfterModel fallback literals called out. That is false. There are multiple production `{ role, parts }` and `{ candidates: ... }` constructions outside `MessageConverter`, including some in files that do not import `@google/genai`. This matters because the stated purpose is to prevent the #2424 structural repeat: a planner relying on §2A.4 would miss non-import structural Google-content currency and would under-spec the enforcement gate.

**Evidence:**

- `packages/agents/src/core/DirectMessageProcessor.ts:677-699` builds a blocking BeforeModel synthetic `GenerateContentResponse` with `candidates: [{ content: { role: 'model', parts: [...] } }]`.
- `packages/agents/src/core/DirectMessageProcessor.ts:775-779` has the AfterModel fallback `{ role: 'model', parts: [] }`, which the document does mention only generically.
- `packages/agents/src/core/streamRequestHelpers.ts:162-169` builds a synthetic response via `candidates: [{ ...candidate, finishReason: FinishReason.STOP }]` in `patchMissingFinishReason`.
- `packages/agents/src/core/streamResponseHelpers.ts:296-301` builds `Content[]` with `{ role: 'model', parts: args.consolidatedParts }` for history recording.
- `packages/agents/src/core/agenticLoop/loopHelpers.ts:100-117` writes cancelled tool history through `agentClient.addHistory({ role: 'model', parts: functionCalls })` and `{ role: 'user', parts: [...] }`.
- `packages/agents/src/core/ConversationManager.ts:272-310` creates filtered model output and empty model-content placeholders via `{ ...content, parts: ... }` and `{ role: 'model', parts: [] }`.
- `packages/agents/src/core/subagentExecution.ts:159-166` and `:186-195` return `Content[]` nudges as `{ role: 'user', parts: [{ text: ... }] }`.
- `packages/agents/src/core/subagentToolProcessing.ts:478-484` returns `{ role: 'user', parts: toolResponseParts }`.
- `packages/agents/src/agents/executor-prompt-builder.ts:47-58` is a production file with no raw `@google/genai` import that structurally edits `parts` on messages typed only as `T extends { parts?: ... }`.

**Required fix:** Replace §2A.4 with a complete structural-literal inventory, separated into: (a) synthetic response envelopes (`candidates`), (b) legacy public/contract `Content` construction (`role` + `parts`), (c) history/write-path `Content` construction, (d) hook fallback/adapters, and (e) generic structural `parts` mutation without raw imports. For each, state whether it disappears with contract neutralization, is a bounded external-wire adapter, or must be retyped to `IContent`/`ContentBlock[]`. Update §8 and §9.1 so the gate and acceptance criteria do not pretend that only `MessageConverter` and two fallbacks need structural checks.

### Major 2 — Direct-message inventory misses the blocking BeforeModel synthetic response path

**Location in doc:** §2B.2 lines 224-233; §3.2 row #14 line 268; §9.1 line 562.

**Problem:** The direct-message section correctly notes `_processDirectResponse` fabricates a response after provider output, but it omits `_buildBlockingSyntheticResponse`, which fabricates a `GenerateContentResponse` before any provider output when a BeforeModel hook blocks. That path is not just another test fixture; it is production behavior and part of the direct-message response contract. A later plan must preserve the hook-block observable behavior while replacing the Google-shaped synthetic response with `ModelOutput`.

**Evidence:** `packages/agents/src/core/DirectMessageProcessor.ts:677-699` defines `_buildBlockingSyntheticResponse(beforeModelResult): GenerateContentResponse` and returns either `beforeModelResult.getSyntheticResponse()` cast as `GenerateContentResponse` or an inline `{ candidates: [{ content: { role: 'model', parts: [{ text: ... }] } }] }` object. `_processDirectResponse` separately fabricates via `convertIContentToResponse(lastResponse)` at `DirectMessageProcessor.ts:744-753`, so there are at least two direct-message synthetic-response sites.

**Required fix:** Add `_buildBlockingSyntheticResponse` to §2B.2, §3.2 row #14, the synthetic-response acceptance criterion, and the hook-boundary table. State the neutral target explicitly: a blocking BeforeModel hook must yield/return a neutral `ModelOutput` (or equivalent neutral hook result) with the same text/reason semantics, not a `GenerateContentResponse` cast or inline candidate envelope.

### Major 3 — Enforcement gate description is too broad in one place and too narrow in another

**Location in doc:** §8 lines 536-546; §9.1 lines 559-561.

**Problem:** The gate is correctly described as needing AST awareness, but its concrete structural-literal rule is not precise enough to be implementable without either false positives or blind spots. It says to match any object literal bearing a `candidates:` array property, or a response/stream-context literal bearing `parts:`. The broad `candidates:` part will false-positive on unrelated domain candidates; the repository already has unrelated candidates such as compression load-balancing and profile selection. The `parts:` rule is simultaneously too narrow because the document's own structural bypass concern includes generic `parts` mutation with no `role` (e.g. executor prompt templating), but the rule only says “response/stream-context” without defining how to identify that context.

**Evidence:**

- Unrelated production `candidates` identifiers exist in `packages/agents/src/core/CompressionLoadBalancingProvider.ts:34`, `packages/agents/src/core/CompressionProfileResolver.ts:401`, and `packages/agents/src/api/control/profilesControl.ts:392`; a naive object-literal `candidates:` ban would catch legitimate non-Gemini code.
- Real structural `parts` manipulation with no raw import exists in `packages/agents/src/agents/executor-prompt-builder.ts:47-58` (`T extends { parts?: ... }`, `content.parts`, `{ ...content, parts: newParts }`), and §2A.4 currently does not inventory it.
- Real Gemini response candidates are present at `MessageConverter.ts:523-540`, `DirectMessageProcessor.ts:684-699`, and `streamRequestHelpers.ts:166-169`.

**Required fix:** Rewrite §8 to specify concrete AST patterns that distinguish Gemini envelopes from unrelated candidates: e.g. `candidates` whose elements contain `content` with `role`/`parts`, `GenerateContentResponse` casts/annotations, imports/type references bound to SDK/Contract payloads, and object literals passed to known Google-shaped APIs (`addHistory`, `setHistory`, `filterHookRestrictedContent`, etc.) until those APIs are neutralized. Add an allow-list strategy for unrelated domain `candidates` and define how the gate handles generic structural `parts` mutators such as `executor-prompt-builder.ts`.

### Major 4 — Usage metadata mapping omits `thoughtsTokenCount`, creating a lossy edge the overview says is covered

**Location in doc:** §5.2 line 436; §5.3 line 452; §7 line 501; §7A lines 508-526.

**Problem:** The substitution table says `thoughtsTokenCount -> reasoningTokens` and that `usageMetadataToUsageStats` already does this map. The current code does not map `thoughtsTokenCount` at all. Since the overview's job is to be an accurate map of current broken architecture and neutral gaps, this should be called out as an existing lossy edge, not stated as already handled.

**Evidence:** `packages/agents/src/core/streamChunkWrapper.ts:43-69` maps `promptTokenCount`, `candidatesTokenCount`, `totalTokenCount`, `cachedContentTokenCount`, `toolUsePromptTokenCount`, `cache_read_input_tokens`, and `cache_creation_input_tokens`, but there is no `thoughtsTokenCount` mapping to `reasoningTokens`. `packages/agents/src/core/googlePartHelpers.ts:25-29` extends usage with `toolUsePromptTokenCount`, `cache_read_input_tokens`, and `cache_creation_input_tokens`, but not `thoughtsTokenCount`.

**Required fix:** Change the table and behavioral-contract sections to say `thoughtsTokenCount -> reasoningTokens` is a required migration/fidelity fix, not an already-implemented mapping. Add it to the lossy-edge/open-question or acceptance list so token accounting tests cover reasoning/thought tokens.

### Major 5 — `toModelStreamChunk(IContent)` is presented as sufficient, but it drops provider metadata needed for AFC unless extended

**Location in doc:** §2.4 line 139; §2B.2 line 233; §5.3 lines 449-451; §6.1 lines 474-478.

**Problem:** The document correctly identifies AFC as an open neutral-home question, but it still states that `toModelStreamChunk(iContent)` already maps `IContent` metadata to the neutral chunk and implies the target can simply use it. In current code, `toModelStreamChunk` maps stop/finish reason, usage, and id only; it does not copy `providerMetadata`. That is a concrete gap because the direct-message path currently reads `automaticFunctionCallingHistory` from `lastResponse.metadata.providerMetadata` and attaches it to the synthetic response.

**Evidence:**

- `packages/core/src/llm-types/modelEnvelope.ts:188-210` sets `rawStopReason`, `finishReason`, `usage`, and `responseId`, then returns; it does not copy `icontent.metadata.providerMetadata` to `ModelOutput.providerMetadata`.
- `packages/core/src/llm-types/modelEnvelope.ts:51-59` defines `providerMetadata?: Record<string, unknown>` on `ModelOutput`, so a neutral slot exists but the helper does not populate it.
- `packages/agents/src/core/DirectMessageProcessor.ts:755-764` reads `lastResponse.metadata?.providerMetadata?.['automaticFunctionCallingHistory']` and writes filtered AFC history onto the synthetic `GenerateContentResponse`.

**Required fix:** Amend §2.4 and §5.3 to explicitly state that `toModelStreamChunk` is sufficient for stop/finish/usage/id today but must be extended or wrapped to preserve `providerMetadata`/neutral AFC if the implementation relies on it. Add a concrete acceptance criterion that provider metadata needed for AFC is not lost when replacing `convertIContentToResponse`.

## Minor findings

### Minor 1 — The raw-import inventory table has at least one incorrect symbol list

**Location in doc:** §3.2 row #20 line 274.

**Problem:** `beforeModelHookDecision.ts` is listed as importing `FinishReason` as a type, but the file imports both `FinishReason` and `GenerateContentResponse` with `import type`. That is technically correct TypeScript syntax, but the overview then treats runtime `FinishReason` uses as only in `MessageConverter.ts`, `streamRequestHelpers.ts`, and `streamResponseHelpers.ts`. If the point of row #20 is disposition, the type-only status matters: `beforeModelHookDecision.ts` patches/inspects response finish reasons but does not import the runtime enum value.

**Evidence:** `packages/agents/src/core/beforeModelHookDecision.ts:7` is `import type { FinishReason, GenerateContentResponse } from '@google/genai';`. Runtime `FinishReason` imports are visible in `MessageConverter.ts`, `streamRequestHelpers.ts`, and `streamResponseHelpers.ts` only.

**Required fix:** Clarify row #20 as `FinishReason` (type-only) and reserve “runtime FinishReason value uses” for the files that actually import the value. This avoids over-scoping the runtime-enum replacement work.

### Minor 2 — The document does not distinguish `systemInstruction` text-only migration from legacy `ContentUnion` support

**Location in doc:** §4.1 lines 346-348; §5.2 line 434; §5.3 lines 446-448.

**Problem:** The overview correctly says `ContractGenerateContentConfig.systemInstruction` is Google-shaped because it accepts `ContractContentUnion`, and it maps neutral settings to `systemInstruction?: string`. But it does not call out a compatibility decision for current agents helpers that accept non-string system instructions and extract text from `Part`/`Content`. This is a small but real planner gap because changing `systemInstruction` to string-only may be intentional, but current code has behavior to preserve or explicitly drop.

**Evidence:** `packages/agents/src/core/baseLlmClient.ts:68` accepts `systemInstruction?: string | Part | Part[] | Content`; `baseLlmClient.ts:328-342` converts non-string system instructions by extracting text blocks. `packages/core/src/llm-types/modelRequest.ts:45-61` defines neutral `ModelGenerationSettings.systemInstruction?: string` only.

**Required fix:** Add an open question or substitution-table note: decide whether non-string legacy `systemInstruction` inputs remain accepted at public/compat boundaries and are normalized to string, or whether the neutral contract intentionally narrows to string with a documented breaking change.

### Minor 3 — `executableCode` / `codeExecutionResult` are present in the Google-shaped response contract but have no target disposition

**Location in doc:** §4.1 lines 347-350; §5.2 lines 420-440.

**Problem:** `ContractGenerateContentResponse` includes `executableCode` and `codeExecutionResult`, and `MessageConverter.convertIContentToResponse` fabricates them as `undefined`. The substitution table folds `GenerateContentResponse` into `ModelStreamChunk` / `ModelOutput` but does not explicitly say these fields are unsupported/deleted, moved to `CodeBlock`, or carried in metadata. A planner will need that decision when neutralizing the contract.

**Evidence:** `packages/core/src/core/clientContract.ts:100-105` defines `executableCode` and `codeExecutionResult` on `ContractGenerateContentResponse`; `packages/agents/src/core/MessageConverter.ts:538-539` fabricates both as `undefined`. There are no other production agents uses (`grep` finds only those two assignments under `packages/agents/src`).

**Required fix:** Add a row or note in §5.2: `executableCode`/`codeExecutionResult` are currently synthetic-undefined and either map to `CodeBlock` if real support is required or are removed from the neutral agent contract as unused legacy fields.

### Minor 4 — Public usage-metadata evidence points partly to the wrong package path

**Location in doc:** §2B.1 line 222; §7A lines 514-516.

**Problem:** The doc cites `turn.ts:221-228` for `ServerUsageMetadataEvent`, but the current definition is in core, not the agents `turn.ts` file. This is easy to misread because there is also `packages/agents/src/core/turn.ts`, whose lines 221-228 are constructor/debug setup, not the event type.

**Evidence:** `packages/core/src/core/turn.ts:221-228` defines `ServerUsageMetadataEvent` with `promptTokenCount`, `candidatesTokenCount`, `totalTokenCount`, and `cachedContentTokenCount`. `packages/agents/src/api/event-types.ts:32-41` and `packages/agents/src/api/event-schema.ts:30-39` define the API shape as the overview says.

**Required fix:** Qualify the citation as `packages/core/src/core/turn.ts:221-228` everywhere, and reserve `packages/agents/src/core/turn.ts` citations for the agent loop implementation.

### Minor 5 — Test inventory grouping is plausible but not reproducible enough for a planner

**Location in doc:** §3.3 lines 312-324; §10 lines 593-595.

**Problem:** The document gives verified aggregate test counts but only representative grouped files, not the sorted 54-file test list. Since tests are a major migration burden and the overview includes sorted production and CLI lists in Appendix A, the absence of the sorted test list makes the test inventory less auditable than the rest of the document.

**Evidence:** The raw count is correct (`grep -rl "@google/genai" packages/agents/src` returns 100; production exclusion returns 46; therefore test/helper count is 54), but §3.3 provides only grouped examples and approximate subcounts such as “~6” and “~10”.

**Required fix:** Add an appendix with the sorted 54 test/helper paths or replace approximate group descriptions with exact lists/counts. This is still an overview, not a task list; an exact inventory is in scope.

## Additional open questions

1. **Blocking-hook neutral result shape:** What exact neutral result replaces `BeforeModelHookOutput.getSyntheticResponse()` when the hook returns a Gemini-shaped synthetic response today? The overview should ask whether hook-provided synthetic responses remain a legacy wire shape converted at the boundary or whether hooks must produce neutral model output.
2. **Generic structural `parts` helpers:** Which production helpers that structurally manipulate `{ parts }` without raw imports (for example `executor-prompt-builder.ts`) are external legacy adapters versus internal currency that must become `IContent`/`ContentBlock[]`?
3. **Provider metadata propagation:** If neutral AFC uses `ModelOutput.providerMetadata`, should `toModelStreamChunk` copy all `IContent.metadata.providerMetadata`, or should AFC be promoted to a first-class `ModelOutput.afcHistory?: IContent[]` to avoid making provider metadata load-bearing in agents?
4. **Reasoning-token fidelity:** Should `thoughtsTokenCount` be migrated from legacy usage metadata into `UsageStats.reasoningTokens` for both stream and direct paths, and should public usage events expose it under a legacy or neutral name?
5. **Legacy system-instruction inputs:** Is accepting `Part`/`Content` system instructions still a public compatibility requirement, or can the neutral contract require string-only `systemInstruction`?
