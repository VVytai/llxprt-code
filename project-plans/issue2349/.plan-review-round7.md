# Review of PLAN-20260707-AGENTNEUTRAL

## Verdict: APPROVE-WITH-FIXES

The plan is much stronger than a name-only source swap: it recognizes structural Google-shape as the problem, stages the synthetic-response deletion, neutralizes `clientContract.ts` cross-package, adds early AST ratcheting, uses a central allow-list rather than inline exemptions, and covers the rejected #2424 vectors in both production and tests. However, it is not yet safe for literal mechanical execution. The most serious blockers are an executable-path hole in the early gate fixture verification, a usage-metadata branch that can break current CLI public-event consumers, several verification commands that are comments rather than hard assertions, and template/TDD evidence gaps that would let a coordinator mark phases complete without proving the claimed behavioral preservation.

## Critical findings

### Critical 1 — P19 option B can break existing CLI/public-event consumers without any owning phase

**Location in plan:** `project-plans/issue2349/plan/18-usage-metadata-characterization-tdd.md:13-17`, `project-plans/issue2349/plan/19-usage-metadata-boundary-impl.md:13-17`, `project-plans/issue2349/plan/19-usage-metadata-boundary-impl.md:71-74`

**Problem:** P18 deterministically selects option B if `done.finished.usageMetadata` currently exposes neutral keys, and P19's option B retypes only the agents API declared types (`UsageMetadataValue` / `FinishedValue.usageMetadata`) and writes no mapper. The plan does not include the CLI/public-event consumers that currently read Gemini-named usage keys from `AgentEvent` usage payloads. If B is selected, those consumers will either fail typecheck or silently lose usage telemetry unless P19 (or a dedicated cross-package phase) migrates them atomically. This is not covered by P21's `clientContract.ts` consumer list, which is `Contract*`-focused, not public usage-event-focused.

**Evidence:**
- P18 selector: neutral observed keys ⇒ option B; Gemini keys ⇒ option C (`plan/18-usage-metadata-characterization-tdd.md:13-17`).
- P19 option B only says to retype `api/event-types.ts` / `event-schema.ts` and write no mapper (`plan/19-usage-metadata-boundary-impl.md:71-74`).
- Current CLI consumers read Gemini-named keys: `packages/cli/src/ui/hooks/agentStream/agentEventDispatcher.ts:401-408` reads `event.usage.promptTokenCount`; `packages/cli/src/zed-integration/zedIntegration.ts:611-615` reads `usage.candidatesTokenCount` / `usage.totalTokenCount`.
- Overview explicitly says option B is a public breaking change requiring coordinated CLI update (`project-plans/issue2349/overview.md:742-748`).

**Required fix:** Make the usage-boundary decision executable cross-package. Either (a) choose option C unconditionally for #2349 to preserve the existing public wire and avoid CLI migration in this plan, or (b) if option B remains possible, add a P19-owned consumer inventory and implementation tasks for every CLI/core/public-event consumer of `promptTokenCount` / `candidatesTokenCount` / `totalTokenCount`, including tests, with a build-green checkpoint. P19's verification must grep and typecheck those consumers, not just agents API files.

### Critical 2 — P02 verification passes fixture file paths, but P02 never specifies that the gate accepts arbitrary file arguments

**Location in plan:** `project-plans/issue2349/plan/02-ast-gate-skeleton.md:124-148`, `project-plans/issue2349/plan/02-ast-gate-skeleton.md:220-235`; pseudocode `project-plans/issue2349/analysis/pseudocode/enforcement-gate.md:6-10`, `:18-23`

**Problem:** P02's hard verification runs `npx tsx scripts/agents-neutral-gate.ts --enforce-imports scripts/__tests__/fixtures/<fixture>.ts`, expecting the gate to evaluate the explicit fixture file. But P02's implementation task says the parser walks `packages/agents/src` production files, excluding tests, and the enforcement pseudocode defines inputs as files under `packages/agents/src` with production/test scopes. Nowhere does P02 require a positional path argument, a `--files` option, or an alternate fixture-root mode. A literal implementer can satisfy the stated implementation contract (walk production files only) and then the P02 fixture verification is non-executable or vacuously ignores the fixture path.

**Evidence:**
- P02 script task: walk `packages/agents/src` production files and exclude test/spec/helper files (`plan/02-ast-gate-skeleton.md:124-148`).
- P02 verification invokes the script with explicit fixture file arguments (`plan/02-ast-gate-skeleton.md:220-235`).
- Enforcement pseudocode inputs are scoped to `packages/agents/src` production/test files, not arbitrary script fixtures (`analysis/pseudocode/enforcement-gate.md:6-10`, `:18-23`).

**Required fix:** Add an explicit CLI contract to the pseudocode and P02/P29/P31 tasks: e.g. `--files <path...>` or positional file arguments override the default scan root while preserving the same AST checks and allow-list semantics. Then update P02/P30 verification to use that documented mode. Alternatively, place fixtures under a temporary fake `packages/agents/src` tree and pass a documented `--root` option. Do not leave fixture-path handling implicit.

## Major findings

### Major 1 — Multiple verification phases still do not satisfy the required expanded GIVEN/WHEN/THEN template

**Location in plan:** examples include `project-plans/issue2349/plan/07a-streamprocessor-impl-verification.md:14-18`, `project-plans/issue2349/plan/29-enforcement-gate-stub.md:39-58`, `project-plans/issue2349/plan/31-enforcement-gate-impl.md:16-35`; governing docs `dev-docs/PLAN-TEMPLATE.md:41-53`, `project-plans/issue2349/plan/verification-template.md:5-13`

**Problem:** The plan claims every NNa verification phase must include explicitly labeled `GIVEN`, `WHEN`, and `THEN` bullets, but several phase files compress requirements into prose or combine multiple `GIVEN`/`THEN` clauses in one sentence. A mechanical coordinator checking template compliance cannot distinguish full behavioral gate text from references/prose summaries.

**Evidence:**
- PLAN-TEMPLATE requires full requirement text plus explicit GIVEN/WHEN/THEN behavior (`dev-docs/PLAN-TEMPLATE.md:41-53`).
- The plan's own verification template mandates three explicitly labeled bullets in every NNa requirement block (`plan/verification-template.md:5-13`).
- P07a has a prose “Confirms REQ…” block and a C3 paragraph, but no three labeled G/W/T bullets for the gate (`plan/07a-streamprocessor-impl-verification.md:14-18`).
- P29 and P31 use compressed behaviors such as “GIVEN a hit… THEN it is exempt; GIVEN the same…” rather than separate labeled bullets for each gate behavior (`plan/29-enforcement-gate-stub.md:52-58`, `plan/31-enforcement-gate-impl.md:24-35`).

**Required fix:** Normalize every NNa and implementation phase requirement section to the exact template form. Each requirement must have separate labeled `- GIVEN:`, `- WHEN:`, `- THEN:` bullets with the specific artifacts/command/pass-fail behavior. Do not rely on `verification-template.md` as a substitute for per-phase expanded text.

### Major 2 — Several “verification commands” remain advisory comments instead of hard failing checks

**Location in plan:** `project-plans/issue2349/plan/07-streamprocessor-impl.md:131-154`, `project-plans/issue2349/plan/07a-streamprocessor-impl-verification.md:22-33`, `project-plans/issue2349/plan/19-usage-metadata-boundary-impl.md:113-119`

**Problem:** Important acceptance checks are written as comments after greps, not shell assertions that fail on violations. This undermines the plan's fraud-prevention model: a coordinator can paste output that contradicts the comment and still proceed unless the verifier manually catches it. The plan fixed some earlier echo-and-continue problems, but not these.

**Evidence:**
- P07 says `grep -nE "getModifiedResponse..." ... # EXACTLY ONE`, `grep ... # NONE`, and `npx tsx ... --count # net AST count STRICTLY LOWER`, but does not mechanically assert one hit, zero hits, or count decrease (`plan/07-streamprocessor-impl.md:131-154`).
- P07a uses checkbox prose for the same checks instead of executable assertions (`plan/07a-streamprocessor-impl-verification.md:22-33`).
- P19 requires P19-owned structural IDs to be absent and count lower, but again leaves it as a comment after `--by-file` / `--count` (`plan/19-usage-metadata-boundary-impl.md:113-119`).

**Required fix:** Replace comment-only expectations with failing shell snippets. Examples: capture grep counts and `test "$count" -eq 1`; fail if forbidden greps return any line; capture previous/current gate count from the baseline artifact and compare numerically; grep `--by-file` output for exact owned site IDs and fail if present. Apply this consistently to every phase with “EXACTLY”, “NONE”, “STRICTLY LOWER”, or “MUST” in a command comment.

### Major 3 — P07 creates a structural hook-wire adapter before the final gate, but the allow-list update is not a concrete P07 task

**Location in plan:** `project-plans/issue2349/plan/07-streamprocessor-impl.md:68-73`, `project-plans/issue2349/plan/07-streamprocessor-impl.md:134-140`, `project-plans/issue2349/plan/31-enforcement-gate-impl.md:43-49`; overview `project-plans/issue2349/overview.md:762-783`

**Problem:** P07 creates `hookWireAdapter.ts`, which intentionally reads hook JSON-wire `candidates/content/parts`. The text says it is “recorded” in the central allow-list, but the P07 implementation task list does not include modifying `dev-docs/agents-neutral-gate-allowlist.md` with the exact AST-context entry. P31 later finalizes that entry, but P07 runs after P02's AST `--count` and before P31. If the P07 adapter is not allow-listed at P07, the early structural count can increase or the slice can fail its own ratchet; if implementers weaken the gate to avoid that, the central allow-list discipline is compromised.

**Evidence:**
- P07 creates `hookWireAdapter.ts` and maps `candidates?.[0]?.content?.parts` from the hook wire (`plan/07-streamprocessor-impl.md:68-73`).
- P07 verifies the adapter exists and contains `afterModelModifiedToChunk`, but does not verify an allow-list entry for it (`plan/07-streamprocessor-impl.md:134-140`).
- P31 says the target allow-list includes `hookWireAdapter.ts` and that P07/P13/P31 converge on the same AST-context rule (`plan/31-enforcement-gate-impl.md:43-49`).
- The overview requires every bounded structural exception to be granted only by a central allow-list entry, not prose or inline comments (`overview.md:762-783`).

**Required fix:** Add `dev-docs/agents-neutral-gate-allowlist.md` to P07's “Files to Modify” and verification. P07 must add the narrow `afterModelModifiedToChunk` AST-context entry when the adapter is introduced; P13 extends it for the additional named functions; P31 only finalizes/tests the matcher. Verification must prove a bare file-level entry is rejected and the P07 count subtracts only the named context.

### Major 4 — P02 overstates “banned symbols from ANY specifier,” conflicting with the overview's binding/provenance rule

**Location in plan:** `project-plans/issue2349/plan/02-ast-gate-skeleton.md:19-31`, `project-plans/issue2349/plan/02-ast-gate-skeleton.md:143-148`; overview `project-plans/issue2349/overview.md:764-766`

**Problem:** P02 describes `checkB_bannedSymbols` as flagging a banned Google symbol “from ANY specifier.” That is too broad if implemented literally: names such as `Content`, `Type`, `Schema`, and `Tool` can be legitimate neutral/domain identifiers when not imported from a banned module or structurally Google-shaped. The overview's gate requirement is more precise: match imports/aliases bound to banned modules or structural redeclarations, not arbitrary import names from any module.

**Evidence:**
- P02 says banned Google symbol imports from “any specifier” catch a re-export bypass (`plan/02-ast-gate-skeleton.md:19-31`, `:143-148`).
- The overview explicitly says not to ban arbitrary local/domain identifiers by name; the gate keys on import binding / structural shape and banned modules (`overview.md:764-766`).

**Required fix:** Rewrite P02/P30/P31 checkB language to match the overview: flag banned symbol names only when imported/aliased from `@google/genai`, `core/clientContract`, `llm-types/geminiContent`, or an identified re-export module that itself re-exports those banned bindings. If “any specifier” is retained for re-export hardening, require resolver logic proving the specifier re-exports a banned binding, and add false-positive fixtures for legitimate neutral `Content`/`Tool`/`Schema` imports from safe modules.

### Major 5 — P28's behavior-coverage preservation gate depends on pre-migration counts that P0.5 never records

**Location in plan:** `project-plans/issue2349/plan/28-test-migration.md:83-104`, `project-plans/issue2349/plan/00a-preflight-verification.md:37-149`

**Problem:** P28 requires post-migration behavior-area test counts to be greater than or equal to “pre-migration values recorded in the completion marker.” But P0.5, the natural place to capture those pre-migration counts, does not include any behavior-area count task. A coordinator reaching P28 will not have an authoritative baseline to compare against, so the preservation gate can become non-executable or be filled in after the fact.

**Evidence:**
- P28 requires pre-migration counts captured at P0.5 and compares post-migration counts against them (`plan/28-test-migration.md:83-104`).
- P0.5 tasks enumerate provider neutrality, neutral types, synthetic fabricators, client contract, importer counts, runtime enums, AST tooling, usage paths, DTO sufficiency, mutation tooling, and gate script names, but no behavior-area test count capture (`plan/00a-preflight-verification.md:37-149`).

**Required fix:** Add a P0.5 task that runs exactly the P28 behavior-area probes and records the baseline counts in `.completed/P0.5.md`. P28 must parse those recorded values mechanically and fail if missing or if any post count drops. Do not allow P28 to invent baselines during the migration.

### Major 6 — P11 intentionally creates a temporary Google-shaped compat island; the constraints are good but the plan must prove it cannot survive past P13 in the gate baseline

**Location in plan:** `project-plans/issue2349/plan/11-side-channel-retirement-impl.md:57-68`, `project-plans/issue2349/plan/11-side-channel-retirement-impl.md:93-102`, `project-plans/issue2349/plan/13-directmessageprocessor-impl.md:107`, `project-plans/issue2349/plan/13-directmessageprocessor-impl.md:171`

**Problem:** The temporary `beforeModelBlockingCompat.ts` is a pragmatic build-order bridge, but it is exactly the kind of structural Google-shaped island this issue is trying to eliminate. The plan adds line-count/export constraints and a P13 deletion task, which is good. What is still missing is a mechanically tied baseline/allow-list invariant: after P13, the P11 allow-list entry and any structurally equivalent helper must be impossible to keep or reintroduce without the gate failing.

**Evidence:**
- P11 creates a temporary `GenerateContentResponse` compat module, allow-listed until P13 (`plan/11-side-channel-retirement-impl.md:57-68`).
- P11 verifies only one export and a line cap (`plan/11-side-channel-retirement-impl.md:93-102`).

- P30 has a fixture for reintroduced compat shape after the entry is removed (`plan/30-enforcement-gate-tdd.md:56-58`), but this is not explicitly tied to P13's deletion marker/baseline update.

**Required fix:** Add a P13 verification command that runs the P30/P31 compat-reintroduction fixture (or an equivalent direct gate fixture) after deleting `beforeModelBlockingCompat.ts`, proving the freed allow-list slot cannot exempt a new helper. Also require P13 to update the gate baseline/allow-list artifact and fail if `beforeModelBlockingCompat` appears anywhere in the final allow-list or `--by-file` output.

## Additional risks / open questions

1. **Mutation cost and report integrity.** P33's mutation coverage is now much better than an API-only run, and the archived per-slice report check is sound in principle (`plan/33-full-verification.md:51-99`). The risk is report freshness: require each mutation report to include the commit SHA or tree hash of the phase it verifies so stale reports cannot satisfy P33 after later edits.
2. **Option B for usage metadata is broader than agents.** Even if typecheck catches the known CLI consumers, downstream users may rely on the Gemini-named public usage wire. If option B remains allowed, the plan should require an explicit breaking-change note and CLI snapshot updates; otherwise prefer option C for compatibility.
3. **The plan is large but logically sliced.** P21 remains the largest atomic phase. Its checkpointed diff review requirement is appropriate (`plan/21-clientcontract-impl.md:73-78`); keep it mandatory and do not split P21 by introducing a dual-typed `Contract*` compatibility surface.

## Summary

The plan is architecturally aligned with issue #2349 and avoids the rejected #2424 source-swap in most important ways: structural AST enforcement, neutral pipeline currency, side-channel retirement, cross-package contract deletion, and behavioral characterization are all present. It still needs fixes before execution: make the P02 fixture mode real, close the P19 option-B CLI/public-consumer hole, convert remaining comment-based verifications into hard assertions, add the missing P0.5 behavior-count baseline, and normalize the required GIVEN/WHEN/THEN phase text. With those changes, the plan should be safe for a coordinator to execute mechanically.
